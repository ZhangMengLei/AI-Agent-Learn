# LLM API 阶段复盘

完成 LLM API 与 SDK 阶段后，你应该从“会和 AI 对话”升级到“能把模型接入自己的程序”。本阶段重点不是记住某个 SDK 的语法，而是理解 LLM 应用的基本工程结构。

## 本阶段你应该掌握什么

你应该能够说明：

- API Key 是什么，为什么不能写进代码仓库。
- `model`、`messages`、`temperature`、`max_tokens`、`stream` 的作用。
- system、user、assistant 三类消息如何组成多轮对话。
- 普通输出和流式输出的区别。
- 为什么要记录调用日志。
- token、上下文长度、延迟和成本之间的关系。
- Rate Limit、鉴权失败、网络失败等常见错误如何处理。

## 一个最小 LLM 应用的结构

可以把 LLM 应用理解成下面流程：

```text
读取配置
  ↓
接收用户输入
  ↓
构造 messages
  ↓
调用模型 API
  ↓
处理响应或错误
  ↓
展示结果
  ↓
保存历史和日志
```

CLI Chatbot 是最小但完整的练习项目。后续的工具调用、RAG、Agent，本质上都会在这个流程上继续扩展。

## 常见错误

### 1. 把真实 API Key 写进代码

问题：一旦提交到仓库，Key 可能泄露并产生费用风险。

正确做法：

- 使用环境变量。
- 只提交 `config.example.json`。
- 把真实配置加入 `.gitignore`。

### 2. messages 结构混乱

问题：把 system、user、assistant 混在一个字符串中，后续难以维护多轮对话。

正确做法：使用结构化消息列表：

```json
[
  { "role": "system", "content": "你是一个学习助手。" },
  { "role": "user", "content": "什么是 Prompt？" }
]
```

### 3. 多轮对话没有保存 assistant 回复

问题：只保存用户问题，不保存模型回答，下一轮模型无法理解完整上下文。

正确做法：每轮请求后同时追加 user message 和 assistant message。

### 4. 上下文无限增长

问题：对话越长，token 成本越高，延迟也可能增加，还可能超过模型上下文限制。

正确做法：

- 设置最大历史轮数。
- 对早期对话做摘要。
- 允许用户 `/clear` 清空上下文。

### 5. 只处理成功响应

问题：真实 API 调用经常遇到网络错误、限流、鉴权失败或超时。

正确做法：至少处理：

- 缺少 API Key。
- 401 / 403 鉴权错误。
- 429 限流错误。
- 5xx 服务端错误。
- 请求超时。

### 6. 流式输出没有保存完整回答

问题：只把片段打印到终端，但没有拼接完整文本，导致对话历史缺失。

正确做法：边打印边追加到 `full_text`，请求结束后保存完整回答。

### 7. 日志记录过多敏感信息

问题：日志有助于排查问题，但也可能泄露用户输入、Key 或隐私数据。

正确做法：

- 不记录 API Key。
- 谨慎记录用户原文。
- 错误日志只保留必要排查信息。

## CLI Chatbot 验收清单

如果你完成了项目实验，请用下面清单验收：

- 程序能在终端启动。
- 缺少 API Key 时有清晰提示。
- 用户输入问题后能得到模型回答。
- 支持至少 2 轮连续追问。
- 支持 `/help`、`/clear`、`/exit`。
- `/clear` 后上下文确实被清空。
- 如果实现 stream，输出结束后仍能保存完整回答。
- 日志包含模型名、耗时、是否成功和错误信息。
- 日志和代码中没有真实 API Key。
- README 中写明了运行方式和配置方式。

## 复盘问题

请用自己的话回答下面问题：

1. 一次 API 调用中，`messages` 为什么比单个 prompt 字符串更适合多轮对话？
2. system message 应该放什么？不应该放什么？
3. temperature 调高和调低分别适合哪些场景？
4. max_tokens 设置过大或过小分别有什么问题？
5. 流式输出为什么能改善用户体验？
6. 如何避免上下文越来越长？
7. 遇到 Rate Limit 时，你的程序应该怎么提示或重试？
8. 如果日志里保存用户输入，可能有什么风险？

## 下一步

下一阶段通常会进入 Tool Use 或 RAG。你可以在 CLI Chatbot 的基础上继续扩展：

- 让模型调用工具，例如计算器、天气查询、文件读取。
- 接入本地知识库，实现文档问答。
- 给长对话增加摘要记忆。
- 加入 Prompt 模板库，让用户选择不同任务模式。
- 增加评测脚本，比较不同 Prompt 和参数的效果。

在进入下一阶段前，请确保你已经真正理解：LLM API 不是只发一个字符串，而是一个包含配置、上下文、响应处理、安全和成本控制的工程系统。
