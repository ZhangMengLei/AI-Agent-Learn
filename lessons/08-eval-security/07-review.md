# 08 Eval / Security 阶段复盘

本阶段学习重点是：让 AI 应用从“能回答”走向“可评测、可监控、可复盘、可控风险”。

## 你应该掌握的能力

完成本阶段后，请确认自己能够：

- 解释 golden dataset 的价值：固定测试样例，让模型效果可比较、可回归。
- 为回答质量设计评分规则，而不是只凭感觉判断。
- 记录 token、延迟、错误、风险标签等关键日志。
- 区分正常失败、幻觉、安全失败和系统错误。
- 识别 prompt injection、jailbreak、敏感信息泄露、越权工具调用等风险。
- 生成一份能指导改进的 Markdown 评测报告。

## 常见错误

### 1. 测试集只包含简单问题

错误表现：测试集中都是“什么是 Prompt”“什么是 RAG”这类基础问答。

问题：平均分会虚高，无法发现边界场景、安全攻击和幻觉问题。

改进方式：加入边界、未知、冲突上下文、prompt injection、敏感信息请求和高风险工具操作样例。

### 2. expected points 不可评分

错误示例：

```json
{"expected_points":["回答要好"]}
```

问题：不同评审者无法一致打分。

改进方式：写成可检查要点，例如“说明 Agent 可以规划任务并调用工具”“说明工具调用需要权限控制”。

### 3. 只看平均分

错误表现：平均分 4.3 就认为系统可上线。

问题：一个高风险安全失败可能比多个低风险小错误更严重。

改进方式：同时看安全通过率、高风险失败列表、幻觉率、category 分布和失败样例。

### 4. 日志记录过多敏感内容

错误表现：日志保存完整用户输入、真实邮箱、手机号、API Key、cookie。

风险：排查问题时反而制造泄密问题。

改进方式：记录必要元数据，敏感内容脱敏或哈希化。不要记录真实密钥。

### 5. 过度相信 LLM-as-Judge

错误表现：用另一个模型评分后不再人工抽检。

问题：Judge 也可能偏见、误判或被 prompt injection 影响。

改进方式：LLM-as-Judge 用于初筛，关键版本和高风险样例必须人工复核。

### 6. 报告没有行动建议

错误表现：报告只有平均分、通过率、token 数。

问题：开发者不知道下一步应该改 prompt、改数据集、改工具权限，还是改产品边界。

改进方式：报告必须包含失败原因归类和下一步建议。

## 检查清单

### Golden dataset 检查

- [ ] 是否有唯一 `id`？
- [ ] 是否有明确 `category`？
- [ ] 是否有可检查的 `expected_points`？
- [ ] 是否覆盖正常、边界、未知和安全样例？
- [ ] 是否标注 `risk_level`？
- [ ] 是否避免包含真实隐私和真实密钥？

### 评分检查

- [ ] 是否有 0-5 分或通过/失败的明确标准？
- [ ] 是否记录 `hallucination`？
- [ ] 是否记录 `safety_pass`？
- [ ] 是否能解释每个低分原因？
- [ ] 是否对高风险样例人工复核？

### 日志检查

- [ ] 是否记录 request_id、case_id、model、latency、tokens、success、error_type？
- [ ] 是否避免记录真实 API Key、cookie、token？
- [ ] 是否避免保存未脱敏隐私？
- [ ] 是否能根据日志定位失败用例？
- [ ] 是否区分模型错误、系统错误和安全拦截？

### 报告检查

- [ ] 是否包含 run id、模型、prompt 版本？
- [ ] 是否包含平均分、通过率、安全通过率、幻觉率？
- [ ] 是否列出失败用例？
- [ ] 是否列出高风险安全问题？
- [ ] 是否给出下一步改进建议？

## 自测问题

1. 为什么 golden dataset 要长期保存，而不是每次临时写几个问题？
2. 如果平均分上升但安全失败增加，应该如何判断版本质量？
3. 哪些日志字段对排错有用？哪些字段可能造成隐私风险？
4. LLM-as-Judge 适合做什么？不适合做什么？
5. Prompt injection 样例应该如何设计进评测集？

## 下一步

完成本阶段后，你可以继续做三类扩展：

1. **自动化回归**：每次修改 prompt 或检索策略后自动跑一小组核心测试。
2. **多版本对比**：比较不同 prompt、不同模型、不同 RAG 配置的效果。
3. **安全门禁**：把高风险样例设为上线前必须通过的检查项。

最终目标不是追求一个漂亮分数，而是建立一套能持续发现问题、定位原因、指导改进的 AI 工程质量体系。
