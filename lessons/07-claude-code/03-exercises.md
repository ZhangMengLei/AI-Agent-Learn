# Claude Code 练习

这些练习面向中文初学者，但要求你用工程方式思考：任务是否适合 Agent、边界是否清楚、验证是否可重复、安全风险是否被控制。

## 练习 1：识别适合 CLI Agent 的任务

判断下面任务是否适合 Claude Code，并说明哪些需要人工确认。

1. 解释一个函数作用。
2. 修改项目代码并运行测试。
3. 删除远程数据库。
4. 生成 PR 总结。
5. 批量重构变量名。
6. 给线上用户发通知。
7. 检查当前分支是否意外提交了密钥。
8. 在独立 worktree 中尝试一个重构方案。

思考维度：

- 是否需要访问真实外部系统？
- 是否有不可逆后果？
- 是否能用测试、diff、日志或人工 review 验证？
- 是否需要先计划再执行？

## 练习 2：编写 `CLAUDE.md`

为一个项目写 `CLAUDE.md`，至少包含：

- 项目简介。
- 技术栈和目录结构。
- 常用命令。
- 测试方式。
- 代码规范。
- 安全注意事项。
- 最终回复格式。

必须写清楚：

```md
- 不要自动提交 git。
- 不要读取或输出真实密钥。
- 删除文件、安装依赖、git push、修改 CI 配置前必须询问用户。
- 任务结束时报告修改文件、验证命令和未验证事项。
```

## 练习 3：设计 Slash Command

设计一个 `/study-next` 命令，用于学习下一个 AI Agent 主题。

说明：

- 输入是什么：当前阶段、学习者自评、可用时间。
- 执行步骤是什么：读取课程目录、总结已学内容、推荐下一步。
- 输出格式是什么：今日目标、练习清单、自测问题。
- 风险边界是什么：不要自动改课程文件，除非用户明确要求。

加分：再设计一个 `/implement-small-change` 命令，用于小功能开发，要求包含“先计划、再修改、后验证”。

## 练习 4：设计 Skill

为“小型功能开发并补测试”设计一个 Skill。写出 `SKILL.md` 的核心内容：

```md
---
name: safe-feature
description: ...
---

# Safe Feature Skill

## 何时使用
## 不适用场景
## 工作流程
## 工具和权限规则
## 安全边界
## 输出格式
```

重点练习 `description`：它必须让 Agent 知道何时触发，也必须写出哪些情况要停止并询问。

参考触发描述：

```md
description: 当用户要求在当前仓库实现小型功能、修复 bug、补充测试并运行本地验证时使用。若涉及删除文件、安装依赖、访问网络、git commit、git push、生产环境或真实密钥，必须先询问用户。
```

## 练习 5：设计 Hook

设计一个 hook：当文件被修改后自动运行格式化检查。

思考：

1. 什么时候触发？
2. 匹配哪些文件？
3. 执行什么命令？
4. 命令失败时怎么办？
5. 为什么不应该在 hook 里自动 commit 或 push？

教学版示例：

```json
{
  "name": "format-after-edit",
  "event": "PostToolUse",
  "match": {
    "tool": "Edit",
    "path": "src/**/*.js"
  },
  "command": "npm run format -- --check",
  "on_failure": "show_error_and_stop"
}
```

## 练习 6：权限分类

把下面操作分为“允许自动执行 / 需要确认 / 禁止自动执行”，并说明原因。

- 读取 README。
- 搜索代码。
- 查看 git diff。
- 运行单元测试。
- 安装依赖。
- 删除文件。
- `git commit`。
- `git push`。
- `git push --force`。
- 修改 CI 配置。
- 输出 `.env` 中的真实 token。
- 访问生产数据库。

建议原则：

- 低风险、只读、可重复：可以自动允许。
- 改变环境、代码历史或外部系统：需要确认。
- 可能数据丢失、泄密、破坏生产：禁止自动执行。

## 练习 7：Subagents 任务拆分

如果要生成一整套学习文档，可以如何拆给多个子 Agent？

请按主题拆分，并说明每个子 Agent 的职责：

- 课程结构 Agent：检查目录和章节顺序。
- 初学者体验 Agent：检查概念是否易懂。
- 工程深度 Agent：补充 settings、permissions、hooks、worktree 等实践内容。
- 安全审查 Agent：检查是否出现真实密钥、危险命令、生产操作。
- 验收 Agent：检查是否满足任务要求和可运行性。

最后由主 Agent 汇总，解决冲突并输出最终版本。

## 练习 8：Plan Mode 演练

面对下面需求，先写计划，不要直接修改：

```text
请为登录表单增加邮箱格式校验，错误时展示提示文案，并补充测试。不要提交 git。
```

你的计划应包含：

- 要搜索的关键词。
- 要阅读的文件类型。
- 预计修改范围。
- 测试命令。
- 风险点。
- 需要用户确认的问题。

## 练习 9：Worktree 场景判断

判断哪些场景适合使用独立 worktree：

1. 主工作区有未提交修改，但你想让 Agent 尝试另一个方案。
2. 只想解释一段函数。
3. 同时进行两个互不相关的实验性重构。
4. 要在不污染当前分支的情况下生成教学材料。
5. 要直接修复生产环境数据。

写出最终报告应包含哪些路径信息，特别是“需要主会话合并的路径”。

## 练习 10：常见坑修复

下面是一个糟糕的 Skill 描述：

```md
description: 帮我处理项目任务。
```

请把它改成一个更好的描述，要求包含：

- 触发条件。
- 适用任务。
- 不适用任务。
- 高风险操作处理方式。
- 输出要求。
