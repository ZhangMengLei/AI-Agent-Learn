# Claude Code / CLI / Skill 基础

本节把 Claude Code 看成一个“能在工程仓库里行动的 CLI Agent”。它不是单纯聊天机器人，而是能在用户授权范围内读文件、搜索代码、编辑文件、运行命令、调用工具，并把过程沉淀成可复用工作流的工程助手。

## 1. CLI Agent 是什么

CLI Agent 是运行在命令行里的 AI Agent。它通常具备三类能力：

1. **理解上下文**：读取 `README.md`、`CLAUDE.md`、源码、测试、配置和 Git diff。
2. **执行动作**：搜索、编辑、运行测试、查看日志、生成变更说明。
3. **遵守边界**：通过 settings、permissions、hooks、人工确认来限制高风险操作。

适合 Claude Code 的任务一般具有这些特征：

- 需要理解多个文件，而不是只回答一个概念问题。
- 可以通过测试、lint、diff 或人工 review 验收。
- 能明确说明范围和禁止事项。
- 允许 Agent 在隔离环境里尝试、修正、复盘。

不适合直接交给 Agent 自动执行的任务：删除生产数据、发布线上版本、输出真实密钥、绕过权限、未经确认推送代码。

## 2. Claude Code 是什么

Claude Code 是面向软件工程任务的 CLI Agent。它适合做：

- 解释陌生代码库。
- 修复 bug。
- 添加测试。
- 小步重构。
- 代码审查和安全审查。
- 生成迁移计划。
- 自动化重复流程。

一个成熟的 Claude Code 工作方式通常是：

```text
读项目规则 -> 搜索相关代码 -> 阅读最小必要文件 -> 制定计划 -> 等待确认 -> 修改 -> 验证 -> 总结
```

## 3. `CLAUDE.md`：项目级说明书

`CLAUDE.md` 是给 Claude Code 的项目说明。它相当于“新同事入组文档 + 自动化边界说明”。

建议包含：

- 项目介绍和技术栈。
- 常用命令：测试、lint、格式化、启动开发服务。
- 代码规范：命名、目录、测试要求。
- 工作流程：是否需要先计划、是否允许直接修改。
- 安全边界：不要输出密钥、不要自动提交、不要连接生产环境。
- 最终回复格式：修改文件、验证结果、风险和后续建议。

不要只写“请写高质量代码”。Agent 需要的是可执行规则，例如：

```md
- 修改代码前先搜索相关测试。
- 只改与当前任务直接相关的文件。
- 删除文件、git commit、git push、访问生产系统前必须询问用户。
- 任务结束时报告修改文件、运行过的命令和未验证事项。
```

## 4. Slash Command：固定流程入口

Slash Command 是一个可复用的工作流入口，适合把常见任务写成稳定步骤。

示例：

```text
/review
/security-review
/implement-feature
/summarize-work
```

它适合描述“如何做一件事”：输入是什么、步骤是什么、输出格式是什么、遇到风险何时停下来问用户。

## 5. Skill 是什么

Skill 是比单个 slash command 更完整的能力包。它通常封装某个领域或某类任务的知识、流程、工具使用方式和安全约束。

可以把它理解为：

```text
Skill = 触发说明 + 专业知识 + 操作流程 + 工具规则 + 示例/脚手架
```

### Skill 解决什么问题

当任务只靠一句 prompt 容易不稳定时，可以用 Skill 固化经验：

- 同一类任务反复出现，例如 PR review、安全审查、迁移代码、生成发布说明。
- 任务有固定术语、目录、约束和验收标准。
- 需要先检查风险，再决定能否执行。
- 需要调用特定工具或遵守特定顺序。

### Skill 和 slash command 的区别

| 项目 | Slash Command | Skill |
| --- | --- | --- |
| 关注点 | 一条可执行流程 | 一个领域能力包 |
| 典型内容 | 步骤、输出格式 | 触发条件、背景知识、工具规则、流程、示例 |
| 适用场景 | 固定任务入口 | 专业任务、复杂规范、跨项目复用 |
| 例子 | `/review` | `security-review`、`claude-api`、内部文档处理 skill |

### 什么时候应该用 Skill

适合使用 Skill：

- 用户请求命中某个专业领域，例如“做安全审查”“检查 Claude API prompt caching”。
- 项目里有固定流程，需要 Agent 每次都遵守。
- 操作前必须做风险判断，例如权限、密钥、生产环境、外部系统。
- 学习内容较多，单个 prompt 放不下或容易遗漏。

不一定需要 Skill：

- 一次性小问题，例如“解释这个函数”。
- 没有固定流程的开放讨论。
- 只需要普通代码搜索和阅读。

## 6. Skill 的常见目录结构

教学中可以使用下面结构理解 Skill：

```text
.claude/
  skills/
    my-skill/
      SKILL.md
      examples/
        good-request.md
        bad-request.md
      scripts/
        check.sh
      templates/
        report.md
```

核心文件通常是 `SKILL.md`，里面至少要写：

```md
---
name: my-skill
description: 当用户要做某类任务时使用。这里要写清触发词、适用边界和不适用场景。
---

# My Skill

## 何时使用

## 工作流程

## 工具使用规则

## 安全边界

## 输出格式
```

### 触发/描述怎么写

`description` 决定 Agent 何时应该使用这个 Skill。它要具体、可判定、包含边界。

不推荐：

```md
description: 帮助写代码。
```

推荐：

```md
description: 当用户要求为当前仓库做小型功能开发、补测试、运行本地验证并输出变更总结时使用。若涉及删除文件、git commit、git push、访问网络或生产环境，必须先询问用户。
```

好的 description 应包含：

- 触发词：用户说什么时用。
- 任务边界：能做什么、不能做什么。
- 风险处理：哪些情况先停下询问。
- 输出要求：最终报告什么。

## 7. Settings 和 Permissions

Settings 用于配置 Claude Code 行为。Permissions 用于控制工具和命令是否允许自动执行、需要确认或禁止。

常见配置位置可以按教学方式理解为：

```text
.claude/settings.json        # 项目共享配置，适合提交教学占位规则
.claude/settings.local.json  # 个人本地配置，不应提交真实密钥或个人敏感配置
```

权限策略建议按风险分层：

| 分类 | 示例 | 原则 |
| --- | --- | --- |
| 可自动允许 | 读取文件、列目录、搜索代码、运行单元测试、查看 git diff | 低风险、可重复、只读或局部验证 |
| 需要确认 | 安装依赖、修改配置、创建提交、访问网络、长时间构建 | 改变环境、耗时或影响历史 |
| 禁止自动执行 | force push、删除目录、输出密钥、修改生产数据、绕过测试 | 可能造成数据丢失、泄密或线上事故 |

权限不是“越宽越高效”。初学者常犯的错误是为了减少确认，把 `rm -rf`、`git push`、安装依赖和网络访问全部放开。工程实践中应优先保证可控、可复盘。

## 8. Hooks：关键时机的自动检查

Hooks 是在特定事件前后自动执行的命令。它让团队把“每次都应该做的检查”变成自动化。

常见用途：

- 编辑后运行格式化或 lint。
- 总结前运行相关测试。
- 执行高风险命令前做拦截或提醒。
- 任务结束时输出检查清单。

教学版 hook 需要回答三个问题：

1. **何时触发**：例如编辑后、运行命令前、任务结束前。
2. **执行什么**：例如 `npm test`、`python -m compileall`、自定义检查脚本。
3. **失败怎么办**：停止后续自动执行、报告错误、等待用户决定。

不要用 hook 自动做高风险动作，例如删除文件、提交、推送、修改生产数据。Hook 应优先用于检查和提醒，而不是替人做不可逆操作。

## 9. Subagents：把复杂任务拆开

Subagents 可以把复杂任务拆给多个子 Agent 处理。它们适合并行阅读、独立审查、生成不同角度的发现。

典型拆法：

- 代码搜索子 Agent：找入口、调用链、测试。
- 测试子 Agent：判断现有测试覆盖和缺口。
- 安全子 Agent：检查密钥、权限、命令风险。
- 文档子 Agent：整理变更说明和学习材料。

注意：子 Agent 的结果需要主 Agent 汇总和判断，不能把多个发现直接拼接成最终结论。主 Agent 仍要负责范围控制、冲突解决和最终验收。

## 10. Plan Mode：先计划，再执行

Plan Mode 适合复杂改动或高风险任务。它的价值是把“想清楚”放在“动手改”之前。

建议在这些情况下使用：

- 涉及多个模块或多种技术栈。
- 需求不明确，需要先澄清。
- 可能影响测试、构建、权限或发布流程。
- 用户要求“先给方案，不要直接改”。

一个好的计划应包含：

- 需求理解。
- 需要阅读的文件。
- 修改范围。
- 验证方式。
- 风险点和需要用户确认的问题。

## 11. Worktree：隔离任务环境

Worktree 用于在同一仓库中创建隔离工作区，避免影响当前分支和未完成修改。

适合：

- 多个任务并行开发。
- 尝试不确定方案。
- 让 Agent 在独立目录里改动和验证。
- 保留主工作区的干净状态。

使用 worktree 时仍要注意：

- 不要把敏感文件复制进实验目录。
- 不要自动提交或推送，除非用户明确要求。
- 最终报告需要说明“需要主会话合并的路径”。
- 如果检查失败，要报告失败原因，而不是假装通过。

## 12. 常见坑

1. **Skill description 太泛**：导致不该触发时触发，或该触发时没触发。
2. **只写流程，不写边界**：Agent 知道要做什么，却不知道哪些不能做。
3. **settings 权限过宽**：短期减少确认，长期增加误删、误推、泄密风险。
4. **hook 没有失败策略**：检查失败后继续执行，最后输出“已完成”。
5. **Plan Mode 变成空话**：计划没有文件、命令、风险和验收方式。
6. **worktree 改完不报告路径**：主会话不知道从哪里合并。
7. **真实密钥进入示例**：教学材料、settings、日志里都不能出现真实 token、cookie、API Key。

## 13. 安全边界

Claude Code 可以提高工程效率，但必须保留人工监督。建议默认遵守：

- 不读取、不输出、不提交真实密钥。
- 不自动执行 destructive command，例如 `rm -rf`、`git reset --hard`、`git push --force`。
- 不自动访问生产环境或修改生产数据。
- 不绕过测试、hook、review 或权限系统。
- 不把个人本地配置当作项目公共配置提交。
- 对不确定需求先提问，不靠猜测修改大范围代码。

## 一句话总结

Claude Code 把 Agent 能力放进真实工程流程里；Skill、settings、permissions、hooks、subagents、plan mode 和 worktree 则帮助你把这种能力变得可复用、可验证、可控。
