# 07 Claude Code 阶段复盘

本阶段学习重点是：如何把 Claude Code 从“聊天式助手”变成“有项目规则、有工作流、有安全边界的工程助手”。你不仅要知道命令怎么写，还要能解释为什么这样设计、失败时怎么办、哪些操作不能自动化。

## 你应该掌握的能力

完成本阶段后，请确认自己能够：

- 解释 `CLAUDE.md` 的作用：为 Agent 提供项目上下文、命令、规则和边界。
- 设计 slash command：把重复任务沉淀成稳定流程。
- 解释 Skill 是什么：把领域知识、触发条件、工具规则和安全边界封装成能力包。
- 写出可判定的 Skill `description`：说明何时触发、何时不触发、遇到风险如何处理。
- 说明 hook 的价值：在关键时机自动检查、阻断风险或提醒用户。
- 理解 settings/permissions：哪些操作可以自动化，哪些必须确认，哪些禁止。
- 在复杂任务中使用 plan mode：先计划、再执行、最后复盘。
- 判断何时使用独立 worktree，并在最终报告中给出合并路径。
- 知道 subagents 适合并行搜索、审查和整理，但最终判断必须由主 Agent 汇总。
- 知道为什么不能把真实 API Key、密码、cookie、token 写入仓库、settings、Skill 示例或日志。

## 核心概念对照

| 概念 | 解决什么问题 | 初学者要记住什么 |
| --- | --- | --- |
| `CLAUDE.md` | 项目上下文和规则 | 写具体命令、边界和输出格式 |
| Slash command | 固定任务入口 | 写输入、步骤、输出和确认点 |
| Skill | 领域能力包 | 写清触发描述、流程、工具规则、安全边界 |
| Settings | 配置 Agent 行为 | 不要提交真实个人配置和密钥 |
| Permissions | 控制工具/命令权限 | 低风险允许，高风险确认，不可逆禁止 |
| Hooks | 自动检查关键时机 | 失败处理必须明确，不要自动做危险操作 |
| Subagents | 拆分复杂任务 | 子 Agent 给发现，主 Agent 做判断 |
| Plan mode | 先想清楚再改 | 计划要包含文件、命令、风险、验收方式 |
| Worktree | 隔离工作区 | 最终报告绝对路径和合并路径 |

## 常见错误

### 1. 没有先定义项目边界

错误表现：直接让 Agent 改代码，但没有告诉它项目结构、测试命令、禁止事项。

改进方式：先写 `CLAUDE.md`，至少包含项目简介、常用命令、代码规范、安全规则和最终总结格式。

### 2. slash command 只有一句话

错误示例：

```text
帮我实现功能并测试。
```

问题：缺少输入、步骤、输出格式和风险控制。

改进方式：把命令写成流程：理解需求、搜索代码、制定计划、确认、修改、测试、总结。

### 3. Skill description 太泛

错误示例：

```md
description: 帮助处理项目。
```

问题：Agent 不知道什么任务该触发，也不知道哪些情况必须停下。

改进方式：写成可判定描述：

```md
description: 当用户要求在当前仓库实现小型功能、修复 bug、补充测试并运行本地验证时使用。若涉及删除文件、安装依赖、访问网络、git commit、git push、生产环境或真实密钥，必须先询问用户。
```

### 4. hook 只关注自动化，不关注失败处理

错误表现：hook 写了自动格式化，但没有说明命令失败、测试失败或耗时过长时怎么办。

改进方式：每个 hook 都要回答三个问题：何时触发、执行什么、失败后如何处理。默认策略应是“报告错误并停止后续自动执行”。

### 5. 权限设置过宽

错误表现：为了减少确认，把删除、提交、推送、安装依赖、访问网络都设为自动允许。

风险：可能造成数据丢失、泄密、污染环境或错误发布。

改进方式：低风险只读操作可以自动化；会改变环境、代码历史或外部系统的操作必须确认；不可逆或生产写入应禁止自动执行。

### 6. Plan mode 只有空泛列表

错误表现：计划里只写“阅读代码、修改、测试”，没有具体文件、命令、风险和验收方式。

改进方式：计划必须回答：读哪些文件、搜哪些关键词、改哪些范围、跑什么命令、失败怎么办、需要用户确认什么。

### 7. Worktree 改完不报告路径

错误表现：Agent 在独立 worktree 中完成修改，但最终只说“已完成”，主会话不知道从哪里检查和合并。

改进方式：最终报告必须包含 worktree 绝对路径、修改文件、验证结果、需要主会话合并的路径。

### 8. 任务结束没有验收

错误表现：Agent 说“已完成”，但没有测试结果、修改文件、风险说明。

改进方式：要求最终输出至少包含：改了什么、测试是否通过、未验证事项、下一步建议。

## 检查清单

### `CLAUDE.md` 检查

- [ ] 是否说明项目目标和技术栈？
- [ ] 是否列出常用命令？
- [ ] 是否说明测试和格式化方式？
- [ ] 是否写明不要自动提交 git？
- [ ] 是否禁止输出真实密钥和敏感信息？
- [ ] 是否包含任务完成后的总结格式？

### Slash command 检查

- [ ] 是否有清晰输入？
- [ ] 是否有分步骤执行流程？
- [ ] 是否要求先计划再修改？
- [ ] 是否说明哪些情况要询问用户？
- [ ] 是否规定输出格式？

### Skill 检查

- [ ] 是否有 `name` 和具体 `description`？
- [ ] `description` 是否包含触发词和边界？
- [ ] 是否说明何时使用和不适用场景？
- [ ] 是否说明工具和权限规则？
- [ ] 是否说明安全边界？
- [ ] 是否规定输出格式？
- [ ] 是否避免真实 token、API Key、cookie、个人路径？

### Hooks 检查

- [ ] 是否说明触发事件？
- [ ] 是否说明匹配范围？
- [ ] 是否说明执行命令？
- [ ] 是否说明失败处理？
- [ ] 是否避免执行删除、推送、生产写入等高风险动作？

### Settings / Permissions 检查

- [ ] 是否只使用教学占位符？
- [ ] 是否没有真实 API Key？
- [ ] 是否区分允许、询问、禁止？
- [ ] 是否避免把 destructive command 加入自动 allowlist？
- [ ] 是否把 `git push --force`、`rm -rf`、生产写入列为禁止或人工处理？

### Plan Mode 检查

- [ ] 是否复述需求和限制？
- [ ] 是否列出要阅读的文件和搜索关键词？
- [ ] 是否说明修改范围？
- [ ] 是否说明验证命令？
- [ ] 是否说明风险和需要用户确认的问题？

### Worktree 检查

- [ ] 是否说明为什么需要 worktree？
- [ ] 是否没有复制或提交敏感文件？
- [ ] 是否报告 worktree 绝对路径？
- [ ] 是否报告需要主会话合并的路径？
- [ ] 是否报告验证结果和未验证事项？

## 自测问题

1. 为什么 `CLAUDE.md` 不应该只写“请写高质量代码”？
2. slash command 和普通 prompt 的区别是什么？
3. Skill 和 slash command 的区别是什么？
4. 一个好的 Skill `description` 应包含哪些信息？
5. 一个 hook 如果命令失败，Agent 应该继续执行还是停下来？为什么？
6. 为什么 `git push`、删除文件、修改生产数据不能默认自动执行？
7. settings 里为什么不能写真实 token 或个人隐私配置？
8. 什么时候应该使用 plan mode？
9. 什么时候应该使用独立 worktree？
10. subagents 的结果为什么需要主 Agent 统一判断？
11. 任务完成总结中应该包含哪些信息？

## 最终验收任务

请用自己的话设计一个“安全的小功能开发 Skill”，并说明：

- 它何时触发。
- 它不适合处理哪些任务。
- 它会按什么流程执行。
- 它允许、询问、禁止哪些操作。
- 它如何使用 plan mode。
- 它何时建议使用 worktree。
- 它最终如何汇报修改文件和验证结果。

如果你能让同学根据你的设计复现同样流程，说明你已经掌握了本阶段核心能力。
