# 07 Claude Code 阶段复盘

本阶段学习重点是：如何把 Claude Code 从“聊天式助手”变成“有项目规则、有工作流、有安全边界的工程助手”。

## 你应该掌握的能力

完成本阶段后，请确认自己能够：

- 解释 `CLAUDE.md` 的作用：为 Agent 提供项目上下文、命令、规则和边界。
- 设计 slash command：把重复任务沉淀成稳定流程。
- 说明 hook 的价值：在关键时机自动检查、阻断风险或提醒用户。
- 理解 settings/permissions：哪些操作可以自动化，哪些必须确认。
- 在复杂任务中先计划、再执行、最后复盘。
- 知道为什么不能把真实 API Key、密码、cookie、token 写入仓库。

## 常见错误

### 1. 没有先定义项目边界

错误表现：直接让 Agent 改代码，但没有告诉它项目结构、测试命令、禁止事项。

改进方式：先写 `CLAUDE.md`，至少包含项目简介、常用命令、代码规范、安全规则。

### 2. slash command 只有一句话

错误示例：

```text
帮我实现功能并测试。
```

问题：缺少输入、步骤、输出格式和风险控制。

改进方式：把命令写成流程：理解需求、搜索代码、制定计划、确认、修改、测试、总结。

### 3. hook 只关注自动化，不关注失败处理

错误表现：hook 写了自动格式化，但没有说明命令失败、测试失败或耗时过长时怎么办。

改进方式：每个 hook 都要回答三个问题：何时触发、执行什么、失败后如何处理。

### 4. 权限设置过宽

错误表现：为了减少确认，把删除、提交、推送、安装依赖、访问网络都设为自动允许。

风险：可能造成数据丢失、泄密、污染环境或错误发布。

改进方式：低风险只读操作可以自动化；会改变环境、代码历史或外部系统的操作必须确认。

### 5. 任务结束没有验收

错误表现：Agent 说“已完成”，但没有测试结果、修改文件、风险说明。

改进方式：要求最终输出至少包含：改了什么、测试是否通过、未验证事项、下一步建议。

## 检查清单

### CLAUDE.md 检查

- [ ] 是否说明项目目标和技术栈？
- [ ] 是否列出常用命令？
- [ ] 是否说明测试和格式化方式？
- [ ] 是否写明不要自动提交 git？
- [ ] 是否禁止输出真实密钥和敏感信息？
- [ ] 是否包含任务完成后的总结格式？

### Slash command 检查

- [ ] 是否有清晰输入？
- [ ] 是否有分步骤执行流程？
- [ ] 是否要求先计划再修改？
- [ ] 是否说明哪些情况要询问用户？
- [ ] 是否规定输出格式？

### Hooks 检查

- [ ] 是否说明触发事件？
- [ ] 是否说明匹配范围？
- [ ] 是否说明执行命令？
- [ ] 是否说明失败处理？
- [ ] 是否避免执行删除、推送、生产写入等高风险动作？

### Settings 检查

- [ ] 是否只使用教学占位符？
- [ ] 是否没有真实 API Key？
- [ ] 是否区分允许、询问、禁止？
- [ ] 是否避免把 destructive command 加入自动 allowlist？

## 自测问题

1. 为什么 `CLAUDE.md` 不应该只写“请写高质量代码”？
2. slash command 和普通 prompt 的区别是什么？
3. 一个 hook 如果命令失败，Agent 应该继续执行还是停下来？为什么？
4. 为什么 `git push`、删除文件、修改生产数据不能默认自动执行？
5. 任务完成总结中应该包含哪些信息？
