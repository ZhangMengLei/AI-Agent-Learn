# Tool Use 基础

## Tool Use 是什么

Tool Use 是让模型在需要时调用外部工具，而不是只依赖自己生成答案。

模型本身不会真正查数据库、访问天气接口或读取文件。它会根据工具说明生成一次工具调用请求，真正执行工具的是你的程序。

## Function Calling 是什么

Function Calling 是 Tool Use 的常见实现形式。

你把可调用函数告诉模型，包括：

- 函数名称
- 函数用途
- 参数 schema
- 参数含义
- 返回结果格式

模型判断需要调用函数时，会生成结构化参数。

## 基本流程

```text
用户提问
  ↓
模型判断是否需要工具
  ↓
模型生成工具名称和参数
  ↓
程序执行真实工具
  ↓
程序把工具结果交回模型
  ↓
模型基于结果回答用户
```

## 工具定义要包含什么

一个好工具定义至少包含：

- 名称清晰
- 功能单一
- 描述明确
- 参数结构简单
- 参数约束清楚
- 返回值稳定

## JSON Schema

JSON Schema 用于描述工具参数结构。

例如天气工具需要城市名：

```json
{
  "type": "object",
  "properties": {
    "city": {
      "type": "string",
      "description": "要查询天气的城市名"
    }
  },
  "required": ["city"]
}
```

## 工具选择

模型根据用户意图和工具描述选择工具。

工具描述越清晰，模型越容易选对。

不好的描述：

```text
查询信息
```

更好的描述：

```text
根据城市名称查询当前天气，适合回答天气、温度、降雨相关问题。
```

## 权限确认

不是所有工具都应该自动执行。

可以自动执行：

- 查询天气
- 计算数学表达式
- 读取公开资料

需要确认：

- 删除文件
- 修改数据库
- 发送消息
- 创建订单
- 执行 shell 命令

## 安全风险

Tool Use 的风险来自模型可以影响真实世界动作。

常见风险：

- 参数被用户恶意构造
- 工具执行越权操作
- 工具结果里包含 Prompt Injection
- 模型把不可信工具结果当成指令
- 高风险工具缺少人工确认

## 一句话总结

Tool Use 的本质是：模型负责判断和生成参数，程序负责执行和控制权限。
