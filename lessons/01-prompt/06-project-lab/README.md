# Prompt 项目实现工程说明

本项目实验要求你实现一个“Prompt 模板库”。它不一定需要写成程序，但必须像工程项目一样组织模板、示例、测试记录和验收标准。

## 项目目标

构建一套可复用、可测试、可迭代的 Prompt 模板，帮助初学者理解：

1. Prompt 也需要工程化管理。
2. 好的 Prompt 不是一次写成的，而是通过测试不断改进。
3. 模板应该包含适用场景、输入要求、输出格式和边界说明。
4. 对稳定输出的要求越高，Prompt 越需要结构化。

## 推荐文件结构

```text
04-project-lab/
  README.md
  prompt-library/
    summary.md
    translation.md
    code-explain.md
    requirement-analysis.md
    json-extraction.md
    sql-generation.md

  test-cases/
    summary-cases.md
    code-explain-cases.md
    json-extraction-cases.md
  evaluation/
    scorecard.md
    iteration-log.md
```

目录说明：

- `prompt-library/`：存放最终可复用 Prompt 模板。
- `test-cases/`：存放测试输入和期望观察点。
- `evaluation/scorecard.md`：记录评分标准。
- `evaluation/iteration-log.md`：记录每次修改 Prompt 的原因和效果。

如果你暂时不想创建这么多文件，也可以先用一个 Markdown 文件完成最小版本，但内容要包含模板、测试和复盘。

## 模板文件建议格式

每个 Prompt 模板建议使用统一结构：

```text
# 模板名称

## 适用场景
说明这个 Prompt 适合解决什么问题。

## 不适用场景
说明哪些情况不应该使用它。

## 输入要求
说明用户需要提供哪些信息。

## Prompt 模板
写出可复制使用的 Prompt。

## 输出格式
说明期望模型按什么格式回答。

## 示例输入
给出一段测试输入。

## 示例输出
保存一次较理想的模型输出。

## 常见失败情况
记录模型可能出错的地方。
```

## 必做模板

### 1. 文章总结模板

必须支持：

- 一句话总结。
- 3 到 5 条核心要点。
- 适合人群。
- 行动建议。
- 原文没有的信息要标为“不确定”。

### 2. 翻译模板

必须支持：

- 指定目标语言。
- 保留术语或专有名词。
- 说明语气，例如正式、自然、口语化。
- 对歧义词给出备注。

### 3. 代码解释模板

必须支持：

- 解释整体作用。
- 解释关键逻辑。
- 指出潜在 bug 或易错点。
- 面向初学者说明。

### 4. 需求分析模板

必须支持：

- 用户目标。
- 使用场景。
- 核心功能。
- 非功能要求。
- 边界情况。
- 风险与疑问。

### 5. JSON 信息提取模板

必须支持：

- 严格输出 JSON。
- 字段缺失用 `null`。
- 不输出解释文本。
- 明确 JSON 字段含义。

### 6. SQL 生成模板

必须支持：

- 输入表结构。
- 输入查询目标。
- 输出 SQL。
- 说明可能的性能风险。
- 不凭空编造不存在的表和字段。

### 7. 测试用例生成模板

必须支持：

- 从需求生成测试用例。
- 覆盖正常场景、异常场景和边界场景。
- 输出表格。
- 标注优先级。

## 实现步骤

### 第 1 步：确定模板清单

先列出你要做的模板，不要一开始追求完美。建议先完成 5 个核心模板，再扩展更多模板。

### 第 2 步：写最小可用版本

每个模板先写出最简单版本，确保能完成任务。不要一开始写太长，否则很难判断哪些指令真正有效。

### 第 3 步：准备测试用例

每个模板至少准备 3 类输入：

1. 正常输入：信息完整、表达清楚。
2. 缺失输入：缺少关键信息。
3. 干扰输入：包含无关信息或歧义信息。

### 第 4 步：测试并记录问题

每次测试都记录：

```text
模板名称：
模板版本：
测试输入：
模型输出：
是否符合格式：
是否编造信息：
主要问题：
改进计划：
```

### 第 5 步：迭代模板

常见改进方式：

- 增加输出格式。
- 增加“不知道就说不知道”。
- 增加字段缺失处理规则。
- 增加一个好示例。
- 把复杂任务拆成多步。

### 第 6 步：形成最终模板库

把测试效果较好的版本整理到 `prompt-library/`，并保留测试记录，方便后续继续迭代。

## 验收方式

你可以按下面清单自查：

- 至少完成 5 个 Prompt 模板。
- 每个模板都有适用场景和输入要求。
- 每个模板都有明确输出格式。
- 每个模板至少测试 3 组输入。
- JSON 类模板能够稳定输出合法 JSON。
- 总结类模板不会明显编造原文没有的信息。
- 代码解释类模板适合初学者阅读。
- 模板修改有迭代记录。

## 评分建议

| 维度 | 说明 | 分值 |
| --- | --- | --- |
| 完整性 | 是否覆盖要求的模板类型 | 30 |
| 稳定性 | 输出格式是否稳定 | 25 |
| 可复用性 | 模板是否容易被别人复制使用 | 20 |
| 边界控制 | 是否处理缺失信息和不确定信息 | 15 |
| 复盘质量 | 是否记录问题和改进过程 | 10 |

## 进阶挑战

如果你已经完成基础项目，可以继续尝试：

1. 为每个模板增加一个 Few-shot 示例。
2. 为 JSON 模板增加字段校验说明。
3. 把模板做成命令行菜单，但不要求接入真实 API。
4. 对比不同模型在同一模板下的输出差异。
