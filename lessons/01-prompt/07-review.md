# Prompt 阶段复盘

完成 Prompt Engineering 阶段后，请不要只看“模型有没有回答”，而要重点复盘：你的 Prompt 是否能稳定、可控、可复用地得到目标结果。

## 本阶段你应该掌握什么

你应该能够解释并使用以下概念：

- System Prompt / Developer Prompt / User Prompt 的区别。
- Zero-shot 和 Few-shot 的适用场景。
- 角色、任务、背景、约束、示例、输出格式的作用。
- 为什么结构化输出需要更明确的格式要求。
- 为什么模型会编造，以及如何降低编造概率。
- 为什么复杂任务适合拆成多步 Prompt Chaining。

## 一个好 Prompt 的基本结构

可以用下面结构自查：

```text
角色：你希望模型扮演谁？
任务：你要模型完成什么？
背景：模型需要知道哪些上下文？
输入：用户会提供什么材料？
约束：哪些事情不能做？哪些边界要遵守？
格式：输出应该长什么样？
示例：有没有可参考的输入和输出？
```

并不是每个 Prompt 都必须包含所有部分。简单任务可以简单写，复杂任务才需要更完整的结构。

## 常见错误

### 1. 任务描述过于模糊

错误示例：

```text
帮我优化一下。
```

问题：模型不知道优化什么、面向谁、输出什么。

改进方向：说明目标、对象、标准和输出格式。

### 2. 只给任务，不给背景

错误示例：

```text
写一个学习计划。
```

问题：学习者基础、学习时间、目标岗位都不清楚。

改进方向：补充用户背景、约束条件和阶段目标。

### 3. 想要结构化结果，却没有规定格式

错误示例：

```text
提取姓名、职业和目标。
```

问题：模型可能输出自然语言、列表或 JSON，格式不稳定。

改进方向：明确要求输出 JSON、字段名、缺失值处理方式。

### 4. 没有限制模型不要编造

错误示例：

```text
总结这篇文章，并补充作者观点。
```

问题：如果原文没有作者观点，模型可能自行推断。

改进方向：要求“只能依据原文；原文没有的信息标为不确定”。

### 5. 一次性要求太多

错误示例：

```text
分析需求、设计架构、写代码、写测试、给上线方案。
```

问题：任务跨度太大，输出容易浅、乱、漏。

改进方向：拆成多步，每一步只完成一个明确任务。

### 6. 没有测试不同输入

问题：一个 Prompt 在单个例子上表现好，不代表它稳定。

改进方向：至少测试正常输入、缺失输入、干扰输入三类场景。

## 检查清单

完成本阶段后，请逐项检查：

- 我能把模糊需求改写成清晰 Prompt。
- 我的 Prompt 明确说明了任务目标。
- 我的 Prompt 提供了必要上下文。
- 我的 Prompt 有明确输出格式。
- 我的 Prompt 对缺失信息有处理规则。
- 我的 Prompt 对“不确定”有表达要求。
- 我至少做过一次同任务多 Prompt 对比实验。
- 我知道 Few-shot 示例什么时候有帮助。
- 我知道复杂任务可以用 Prompt Chaining 拆解。
- 我保留了 Prompt 版本和测试记录。

## 复盘问题

请用自己的话回答下面问题：

1. 在你写过的 Prompt 中，哪一个输出最稳定？为什么？
2. 哪一个 Prompt 最容易让模型编造？你如何改进？
3. 角色设定对输出有什么影响？有没有过度设定的情况？
4. 输出格式要求越详细越好吗？什么时候会适得其反？
5. Few-shot 示例比单纯规则更有效的场景是什么？
6. 如果要把 Prompt 交给别人复用，还需要补充哪些说明？
