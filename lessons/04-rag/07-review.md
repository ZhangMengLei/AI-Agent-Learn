# RAG 阶段复盘

这一阶段的核心是让模型基于本地知识回答问题，并能说明答案来自哪里。RAG 的质量不是只由大模型决定，而是由文档质量、chunk 策略、检索效果、Prompt 约束和引用格式共同决定。

## 你应该掌握的内容

完成本阶段后，请确认自己已经理解：

- RAG 的基本流程：ingest → chunk → index → retrieve → generate → cite。
- 为什么要切分 chunk：让检索粒度适中，并控制上下文长度。
- metadata 的价值：让每个片段可追溯到文档、章节和位置。
- 检索不是“找到文档”，而是“找到能回答问题的片段”。
- 引用不是装饰，而是验证答案可信度的入口。
- 当资料不足时，正确回答“不知道”比编造更重要。

## 常见错误

### 1. chunk 太大或太小

chunk 太大：

- 检索命中后包含大量无关内容。
- 容易浪费上下文窗口。
- 模型难以定位答案。

chunk 太小：

- 语义不完整。
- 标题和正文分离。
- 检索结果缺少必要上下文。

建议：先用 500 到 800 个中文字符作为起点，再根据检索结果调参。

### 2. 丢失 metadata

只保存 chunk 文本，不保存来源，会导致无法引用，也无法排查错误。

每个 chunk 至少应保存：

- `chunk_id`
- `document_id`
- `source`
- `title`
- `section`

进阶时可增加 `start_line` 和 `end_line`。

### 3. 检索结果不调试

很多 RAG 错误看起来像“模型答错了”，实际原因是检索结果就不相关。

建议提供 `inspect` 命令或调试输出，至少显示：

- query
- top_k
- score
- chunk_id
- source
- content preview

### 4. Prompt 没有限制资料边界

如果 Prompt 只写“请回答问题”，模型可能使用自身知识补全，导致答案看似合理但无法从文档验证。

建议明确写入：

```text
只能基于检索资料回答。
资料不足时回答“根据当前资料无法确定”。
不要使用资料外知识。
回答必须包含引用。
```

### 5. 引用和答案不对应

常见表现：

- 答案引用了 A 文档，但内容来自 B 文档。
- 引用列表包含没有用到的来源。
- 每句话都没有可追溯依据。

建议：让引用编号和检索片段编号一致，并在回答中只引用实际使用的片段。

### 6. 知识库外问题被强行回答

例如知识库只包含 RAG 教程，用户问“今天上海天气怎么样”，系统却编造天气。

正确做法：当检索结果为空或分数低于阈值时，直接回答资料不足，并建议用户补充文档或换个问题。

## 检查清单

### Ingest

- [ ] 只扫描指定知识库目录。
- [ ] 支持读取多篇 Markdown。
- [ ] 能重复运行并生成稳定索引。
- [ ] 文档更新后可以重新生成索引。

### Chunk

- [ ] chunk 大小适中。
- [ ] 长段落会被二次切分。
- [ ] 相邻 chunk 有必要的 overlap。
- [ ] 每个 chunk 都保存 metadata。

### Retrieval

- [ ] 能返回 Top K 结果。
- [ ] 检索结果包含分数和来源。
- [ ] 能通过 inspect/debug 查看检索片段。
- [ ] 对低相关结果有阈值或兜底策略。

### Generation

- [ ] Prompt 明确限制只能使用资料回答。
- [ ] 空检索或低置信度时不会编造。
- [ ] 回答包含引用。
- [ ] 引用能对应到具体文档和章节。

### 工程质量

- [ ] 配置项集中管理，例如 chunk_size、overlap、top_k。
- [ ] 不在代码中保存真实 API Key。
- [ ] 索引文件可删除后重建。
- [ ] 有一组固定问题用于回归测试。

## 复盘问题

1. 你的 chunk 策略更适合教程文档、API 文档还是 FAQ？为什么？
2. 当检索结果不相关时，你如何判断是 chunk 问题还是检索算法问题？
3. 你的回答是否每个关键结论都能找到来源？
4. 如果用户问跨文档问题，你的系统能否组合多个 chunk？
5. 如果知识库没有答案，你的系统会如何表现？
