# 05 Agent 阶段复盘

本阶段的核心不是“让模型看起来更聪明”，而是学会把大模型放进一个可控的执行系统里。一个可靠的 Agent 必须有明确目标、可观察状态、受限工具、停止条件和可追踪日志。

## 本阶段你应该掌握什么

完成 05-agent 后，你应该能够说明：

1. Chatbot 是一次问答，Agent 是围绕目标持续执行的循环。
2. ReAct 的关键是 `Reason + Act`，工程上常落到 `plan -> act -> observe`。
3. `max_iterations` 是 Agent 的安全带，防止无限循环和失控成本。
4. 工具调用日志是调试、复盘、评估和权限审计的基础。
5. 研究助手不是简单总结器，而是会拆任务、找资料、记录依据并生成报告的系统。

## 推荐复盘问题

### 1. 你的 Agent 为什么会停止？

请检查是否同时具备两类停止条件：

- 正常完成：计划完成、结果足够、用户目标满足。
- 安全停止：达到 `max_iterations`、工具连续失败、输入不合法。

如果只有“模型自己说完成”，系统就不够可靠。

### 2. 你的工具调用是否可追踪？

一次工具调用至少应该能回答：

- 第几轮调用？
- 调了哪个工具？
- 输入是什么？
- 输出是否成功？
- 错误是什么？
- 大概耗时多久？

没有日志的 Agent 很难调试，也很难教学演示。

### 3. Plan、Act、Observe 是否真的分开？

常见问题是把所有逻辑都写在一个巨大函数里。初学项目可以简单，但仍建议保留边界：

- Planner：负责拆解任务。
- Executor：负责选择动作。
- Tool：负责执行外部能力。
- Observer：负责记录结果并更新状态。
- Reporter：负责组织最终输出。

边界清晰后，后续才能替换模型、替换工具或增加评估器。

## 常见错误

### 错误 1：无限循环

表现：Agent 一直搜索、一直总结，无法结束。

改法：

- 增加 `max_iterations`。
- 每轮更新 `current_step`。
- 明确 `is_finished()` 判断。
- 达到上限时输出未完成事项。

### 错误 2：工具名由模型任意生成

表现：模型生成一个不存在的工具名，程序直接报错。

改法：

- 使用工具注册表。
- 只允许调用白名单工具。
- 工具不存在时返回结构化错误。

### 错误 3：Observe 被当成最终答案

表现：工具返回一段资料后，系统直接输出，没有整合和判断。

改法：

- 把观察结果写入 `observations`。
- 再由 Reporter 汇总多个观察结果。
- 明确区分“资料片段”和“最终结论”。

### 错误 4：没有记录失败

表现：工具失败后只打印“出错了”，不知道错在哪里。

改法：

- 统一工具返回结构。
- 在 tool log 中记录 `ok=false` 和 `error`。
- 连续失败时安全停止。

### 错误 5：一开始就接入复杂框架

表现：代码很多，但看不清 Agent 主循环。

改法：

- 先用本地模拟工具跑通最小版本。
- 再逐步接入真实搜索、RAG 或外部 API。
- 保持 `plan -> act -> observe` 可见。

## 检查清单

### 概念检查

- [ ] 我能解释 Agent 与 Chatbot 的区别。
- [ ] 我能解释 ReAct 为什么适合工具调用任务。
- [ ] 我能解释 `max_iterations` 的作用。
- [ ] 我能解释工具日志对调试和审计的价值。

### 工程检查

- [ ] 代码中有明确的 Agent 主循环。
- [ ] 有计划生成逻辑。
- [ ] 有工具注册表或等价白名单机制。
- [ ] 每轮执行都有观察结果。
- [ ] 每次工具调用都有日志。
- [ ] 有正常完成和异常停止两类路径。
- [ ] 不包含真实 API Key。

### 输出检查

- [ ] 最终报告包含研究主题。
- [ ] 最终报告包含执行计划。
- [ ] 最终报告包含关键发现。
- [ ] 最终报告能说明资料来源或观察依据。
- [ ] 提前停止时能说明原因和未完成事项。

## 下一步

学完 Agent 后，下一阶段进入 MCP。你可以把 MCP 理解为“让 Agent 标准化使用外部工具和上下文”的协议。

建议带着这些问题学习 06-mcp：

1. 如果 Agent 的工具越来越多，如何统一接入？
2. 如果多个 AI 客户端都要使用同一批工具，如何复用？
3. 工具、资源、提示词模板是否可以由一个标准 Server 提供？
4. 权限、配置和边界应该放在 Agent 内部，还是由外部服务统一管理？

当你能回答这些问题，就能理解为什么 MCP 对 Agent 工程很重要。
