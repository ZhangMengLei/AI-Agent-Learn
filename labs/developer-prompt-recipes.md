# 常见开发场景 Prompt 模板库

这份模板库面向刚开始使用 Claude Code 或其他 AI 编程助手的同学。每个场景都给出可直接复制的中文 Prompt，并补充输出要求和人工检查点。

使用建议：

- 复制模板后，把 `【】` 中的内容替换成你的真实上下文。
- 不要粘贴真实 API Key、密码、cookie、token、生产数据库连接串或用户隐私数据。
- 对高风险操作，例如删除文件、数据库变更、线上发布、权限修改，要先让 AI 给计划，再由人工确认。
- AI 的输出要当作“建议草稿”，不要不看就执行。

## 1. 需求澄清

### 使用时机

当需求描述比较模糊，只知道“大概要做什么”，但缺少边界、输入输出、验收标准时使用。

### Prompt

```text
你是一个有经验的产品与研发协作助手。请帮我澄清下面这个开发需求。

需求背景：
【用 3-5 句话说明业务背景】

当前已知目标：
【写出目前理解的目标】

已有约束：
【技术栈、时间、不能改的模块、兼容要求等；没有就写“暂无”】

请你：
1. 先复述你对需求的理解。
2. 列出需要向产品、设计、后端、测试或运维确认的问题。
3. 区分“必须确认后才能开发”的问题和“可以边做边确认”的问题。
4. 给出一个初版验收标准清单。
5. 不要替我编造不存在的业务规则。
```

### 输出要求

- 用分点列表输出。
- 问题按角色或主题分组。
- 明确标注阻塞项和非阻塞项。
- 验收标准尽量写成可验证的句子。

### 使用后人工检查点

- 检查 AI 是否误解了业务目标。
- 把阻塞问题发给对应负责人确认。
- 确认验收标准是否能被测试或产品直接使用。
- 删除或改写任何不适合项目实际情况的假设。

## 2. 任务拆解

### 使用时机

当需求已经比较明确，但不知道从哪里开始、如何拆分开发步骤、如何安排优先级时使用。

### Prompt

```text
你是一个资深研发项目拆解助手。请基于下面需求，帮我拆成适合逐步完成的开发任务。

需求说明：
【粘贴需求摘要，不要包含敏感数据】

项目背景：
【项目技术栈、主要目录、相关模块】

我希望的交付物：
【例如：一个接口、一个页面、一个脚本、一组测试】

请你输出：
1. 总体实现思路。
2. 按优先级排列的任务清单。
3. 每个任务的输入、输出、完成标准。
4. 可能的风险和依赖。
5. 建议先做的最小可验证版本。

要求：不要直接写代码，先给计划。
```

### 输出要求

- 使用表格或编号列表。
- 每个任务要足够小，最好能在 30-90 分钟内完成。
- 标出哪些任务可以并行，哪些必须串行。
- 给出最小可验证版本，不追求一次做完。

### 使用后人工检查点

- 检查任务是否过细或过粗。
- 确认任务顺序符合项目依赖关系。
- 删除暂时不需要的“锦上添花”任务。
- 与团队同步风险项和外部依赖。

## 3. 代码解释

### 使用时机

当你接手一个陌生文件、函数、类或模块，想快速理解它做什么、怎么流转、有哪些风险时使用。

### Prompt

```text
你是一个耐心的代码讲解老师。请用适合初学者的方式解释下面代码。

代码位置：
【文件路径或函数名】

我想重点理解：
【例如：主流程、数据结构、异常处理、某个函数为什么这样写】

代码内容：
【粘贴代码片段，或说明请读取哪些文件】

请你：
1. 用一句话概括这段代码的作用。
2. 按执行顺序解释主要逻辑。
3. 解释关键变量、函数和数据结构。
4. 指出可能的边界情况和易错点。
5. 给我 3 个自测问题，帮助确认我是否理解。
```

### 输出要求

- 先讲整体，再讲细节。
- 尽量使用通俗类比，但不要牺牲准确性。
- 对复杂逻辑给出简短伪代码或流程描述。
- 不要无依据地评价代码好坏。

### 使用后人工检查点

- 对照源码确认解释是否与实际逻辑一致。
- 运行或阅读相关测试验证关键路径。
- 重点检查 AI 是否漏掉异常分支。
- 把不理解的术语继续追问。

## 4. Bug 定位

### 使用时机

当程序报错、结果不符合预期、线上或本地复现了异常，但还不知道根因在哪里时使用。

### Prompt

```text
你是一个谨慎的 bug 定位助手。请帮我分析下面问题的可能原因和排查步骤。

现象：
【描述报错、异常行为或错误结果】

期望行为：
【描述正确情况下应该发生什么】

复现步骤：
【1. ... 2. ... 3. ...】

相关日志或报错：
【粘贴脱敏后的日志，不要包含 token、cookie、用户隐私】

相关代码位置：
【文件路径、函数名、最近改动；不确定就写“不确定”】

请你：
1. 先列出最可能的 3-5 个原因，并说明依据。
2. 给出从低风险到高风险的排查顺序。
3. 标出需要补充的信息。
4. 如果需要改代码，先给修改思路，不要直接大改。
5. 区分“已确认事实”和“推测”。
```

### 输出要求

- 根因假设要按可能性排序。
- 每个排查步骤要说明如何验证。
- 不要建议直接删除数据、重置环境或跳过校验。
- 对不确定内容明确写“不确定”。

### 使用后人工检查点

- 先验证最小复现，不要一上来改大量代码。
- 确认日志已经脱敏。
- 检查 AI 是否把相关性误当因果。
- 修改后补充回归测试或复现说明。

## 5. 测试补齐

### 使用时机

当已有功能缺少测试、修复 bug 后需要回归测试、重构前想加保护网时使用。

### Prompt

```text
你是一个测试设计助手。请帮我为下面功能补齐测试方案。

功能说明：
【描述功能做什么】

相关代码：
【文件路径、函数名或代码片段】

当前已有测试：
【列出已有测试文件；没有就写“暂无”】

我使用的测试框架：
【例如 pytest、Jest、JUnit；不确定就写“不确定”】

请你输出：
1. 需要覆盖的正常路径。
2. 需要覆盖的边界情况。
3. 需要覆盖的异常路径。
4. 建议新增或修改的测试用例列表。
5. 每个测试的输入、操作步骤、预期结果。

要求：测试数据使用模拟数据，不要使用真实用户信息或真实密钥。
```

### 输出要求

- 按“正常、边界、异常、回归”分类。
- 每个测试用例名称要清晰表达意图。
- 标出哪些测试最优先。
- 如果需要 mock，要说明 mock 的对象和原因。

### 使用后人工检查点

- 检查测试是否真的覆盖核心风险。
- 确认测试不会访问真实外部服务或生产数据。
- 运行测试并确认失败时能提供有用信息。
- 避免只测试实现细节而忽略用户可见行为。

## 6. 重构

### 使用时机

当代码能跑但可读性差、重复多、函数过长、职责混乱，想在不改变行为的前提下改善结构时使用。

### Prompt

```text
你是一个保守的代码重构助手。请帮我评估并规划下面代码的重构。

重构目标：
【例如：减少重复、拆分长函数、提升可读性、降低耦合】

必须保持不变的行为：
【列出接口、输出、兼容性、错误处理等】

相关代码：
【文件路径或代码片段】

已有测试：
【测试文件路径；没有就写“暂无”】

请你：
1. 先指出当前代码的主要可维护性问题。
2. 给出低风险重构步骤，每一步都要可单独验证。
3. 标出重构前应该补的测试。
4. 说明哪些改动不建议现在做。
5. 如果要给代码示例，只给小片段，不要一次性重写全部。
```

### 输出要求

- 强调不改变外部行为。
- 重构步骤要小而可回滚。
- 区分“必要重构”和“可选优化”。
- 不要为了使用新模式而过度设计。

### 使用后人工检查点

- 先确认测试保护足够，再动核心逻辑。
- 每一步重构后运行相关测试。
- 对外接口、配置、数据格式不能无意变化。
- 检查重构是否真的降低复杂度。

## 7. 安全审查

### 使用时机

当代码涉及认证鉴权、输入处理、文件上传、命令执行、数据库访问、第三方回调、日志输出或敏感配置时使用。

### Prompt

```text
你是一个安全审查助手。请对下面变更进行安全风险检查。

变更背景：
【说明为什么改】

涉及范围：
【文件路径、接口、配置、数据流】

代码或设计：
【粘贴相关片段，不要包含真实密钥、真实 token 或隐私数据】

请重点检查：
1. 是否存在敏感信息泄露风险。
2. 是否存在越权、认证绕过或权限校验不足。
3. 是否存在注入风险，例如 SQL、命令、模板、路径等。
4. 是否存在不安全的文件、网络或反序列化操作。
5. 日志、错误信息是否暴露过多细节。
6. 给出风险等级、原因和修复建议。

要求：不要提供攻击利用步骤，只给防护建议和安全编码建议。
```

### 输出要求

- 按高、中、低风险分组。
- 每个风险包含位置、原因、影响、建议。
- 对无法判断的内容列出需要补充的信息。
- 不输出真实攻击载荷或可直接滥用的细节。

### 使用后人工检查点

- 人工复核高风险项，不要只依赖 AI 判断。
- 确认没有把真实密钥或用户数据发给 AI。
- 对安全修复补充测试或检查规则。
- 必要时请团队安全同学复审。

## 8. 性能分析

### 使用时机

当接口慢、页面卡、任务耗时长、资源占用高，或需要在上线前评估性能风险时使用。

### Prompt

```text
你是一个性能分析助手。请帮我分析下面功能的性能瓶颈和优化方向。

功能或接口：
【名称、路径或入口函数】

当前表现：
【耗时、QPS、内存、CPU、页面加载时间等；没有数据就写“暂无量化数据”】

相关代码或流程：
【文件路径、关键函数、调用链】

数据规模：
【例如列表长度、表数据量、并发量；不确定就写“不确定”】

请你：
1. 先指出需要补充哪些性能指标。
2. 基于现有信息推测可能瓶颈，并标明不确定性。
3. 给出排查顺序和观测方法。
4. 给出低风险优化建议。
5. 标出哪些优化可能改变行为或增加复杂度。
```

### 输出要求

- 先测量，再优化。
- 区分前端、后端、数据库、网络、缓存等方向。
- 每个优化建议说明预期收益和潜在代价。
- 不建议盲目加缓存或并发。

### 使用后人工检查点

- 用真实但脱敏的指标验证瓶颈。
- 确认优化不会破坏正确性和一致性。
- 对缓存、批处理、并发改动做回归测试。
- 记录优化前后的对比数据。

## 9. 接口文档

### 使用时机

当需要给前后端、客户端、测试或第三方说明接口如何调用、参数含义、返回结构和错误码时使用。

### Prompt

```text
你是一个接口文档编写助手。请基于下面信息生成清晰的接口文档草稿。

接口名称：
【接口名】

接口用途：
【说明解决什么问题】

请求方式和路径：
【例如 GET /api/example；不要写真实内网地址】

请求参数：
【参数名、类型、是否必填、说明】

响应示例：
【使用模拟数据，不要包含真实用户信息】

错误情况：
【已知错误码或异常场景】

请输出：
1. 接口简介。
2. 请求方法和路径。
3. 请求头说明，如无特殊要求请写“无特殊要求”。
4. 请求参数表。
5. 响应字段表。
6. 成功响应示例和失败响应示例，均使用模拟数据。
7. 调用注意事项。
```

### 输出要求

- 文档语言简洁，适合协作方阅读。
- 参数表包含名称、类型、必填、说明、示例。
- 示例数据必须是模拟数据。
- 对不确定字段标注“需确认”。

### 使用后人工检查点

- 与真实代码或接口定义核对字段。
- 确认示例没有真实手机号、邮箱、身份证、token 等敏感信息。
- 请接口提供方确认错误码和边界行为。
- 文档发布前检查版本和兼容性说明。

## 10. 数据库变更审查

### 使用时机

当需要新增表、改字段、加索引、写迁移脚本、改查询条件或执行数据修复时使用。

### Prompt

```text
你是一个数据库变更审查助手。请帮我审查下面数据库变更方案。

变更目标：
【为什么要改数据库】

数据库类型：
【例如 MySQL、PostgreSQL、SQLite；不确定就写“不确定”】

变更内容：
【DDL、迁移思路或查询改动；不要包含生产连接串】

数据规模和访问特点：
【表大小、读写频率、是否在线业务；不确定就写“不确定”】

回滚方案：
【已有方案；没有就写“暂无”】

请你检查：
1. 字段类型、默认值、非空约束是否合理。
2. 索引是否必要，是否可能造成写入成本或锁表风险。
3. 是否兼容旧代码和灰度发布。
4. 是否需要数据迁移、回填或校验。
5. 是否有回滚方案和验证方案。
6. 输出风险清单和修改建议。
```

### 输出要求

- 按“结构变更、数据迁移、性能、兼容性、回滚”分组。
- 标出高风险操作。
- 对不确定数据规模不要假设，要提示补充。
- 不生成连接生产库的命令。

### 使用后人工检查点

- 请 DBA 或熟悉数据库的同学复审高风险变更。
- 在测试环境验证迁移和回滚。
- 确认没有生产连接串或真实数据样例泄露。
- 上线前确认备份、监控和告警。

## 11. CI 失败排查

### 使用时机

当提交代码后 CI、单元测试、构建、lint、类型检查或部署前检查失败时使用。

### Prompt

```text
你是一个 CI 失败排查助手。请帮我分析下面 CI 失败原因。

失败阶段：
【例如 install、lint、test、build、typecheck、deploy-check】

最近改动：
【简要说明本次改了哪些文件或功能】

失败日志：
【粘贴关键日志，先脱敏，不要包含 token 或内部凭证】

本地是否能复现：
【能/不能/尚未尝试】

请你：
1. 提取日志中的关键错误。
2. 判断最可能的失败原因。
3. 给出本地复现命令或检查步骤；不确定命令时请说明需要查看项目脚本。
4. 给出修复建议。
5. 区分代码问题、环境问题、依赖问题和偶发问题。
```

### 输出要求

- 先定位第一个关键错误，不要被后续连锁错误带偏。
- 给出最小排查步骤。
- 不建议跳过 CI、关闭测试或绕过检查。
- 对需要权限的操作先提示人工确认。

### 使用后人工检查点

- 确认日志已脱敏。
- 在本地或临时环境复现失败。
- 修复后重新运行对应检查，而不是只看代码。
- 如果是偶发问题，记录证据并观察重跑结果。

## 12. PR 描述

### 使用时机

当准备提交 Pull Request，需要清楚说明改了什么、为什么改、如何验证、有哪些风险时使用。

### Prompt

```text
你是一个代码评审协作助手。请帮我根据下面信息生成 PR 描述草稿。

变更背景：
【为什么需要这个 PR】

主要改动：
【列出文件或功能改动摘要】

验证方式：
【运行了哪些测试、手工验证了什么；没有就写“尚未验证”】

风险和影响范围：
【影响模块、兼容性、回滚方式；不确定就写“不确定”】

请输出一个 PR 描述，包含：
1. Summary：1-3 条要点。
2. Test Plan：列出已执行或待执行的验证项。
3. Risk：说明风险、影响范围和回滚思路。
4. Reviewer Notes：提醒评审重点关注什么。

要求：不要夸大已完成的测试；没有做的测试要明确写“未执行”。
```

### 输出要求

- 语言简洁，方便评审者快速理解。
- 已验证和未验证分开写。
- 风险不要写“无”除非确实评估过。
- 不包含敏感链接、密钥或内部凭证。

### 使用后人工检查点

- 核对 PR 描述是否准确覆盖实际 diff。
- 确认测试项都是真实执行过的。
- 补充截图、日志或说明时先脱敏。
- 标出需要重点评审的复杂逻辑。

## 13. 学习某段代码

### 使用时机

当你不仅想知道代码“做什么”，还想通过它学习设计思路、编码习惯、可改进点和相关知识时使用。

### Prompt

```text
你是一个代码学习教练。请带我学习下面这段代码。

我的水平：
【例如：刚学 Python、会写基础接口、第一次看这个框架】

学习目标：
【例如：理解异步、学习项目结构、掌握测试写法】

代码位置或片段：
【文件路径、函数名或代码片段】

请你：
1. 先用初学者能懂的话解释这段代码解决的问题。
```

- 小练习要安全、可回滚、适合当前水平。

### 使用后人工检查点

- 亲自运行或阅读代码验证理解。
- 完成小练习前先保存当前改动。
- 不要把学习建议直接当成项目改造计划。

## 15. 让 Agent 使用工具

### 使用时机

当你希望 Agent 有序使用读取、搜索、编辑、测试等工具，并且避免越权、误改或过度操作时使用。

### Prompt

```text
你是一个受控的开发 Agent。请使用工具完成下面任务，但必须遵守工具使用边界。

任务：
【描述具体任务】

允许的操作：
【例如：读取项目文件、搜索关键词、只编辑 labs 目录下的 Markdown、运行轻量本地检查】

禁止的操作：
【例如：不调用网络、不提交代码、不修改配置、不删除文件、不写入真实密钥】

工作要求：
1. 开始前先检查相关目录和现有文件。
2. 修改前先说明将要修改的路径。
3. 只做完成任务必要的最小改动。
4. 修改后运行轻量检查。
5. 最后报告修改路径、检查结果和需要人工复核的点。

如果发现任务需要超出允许范围，请暂停并向我确认。
```

### 输出要求

- Agent 应该报告关键工具操作和结果。
- 输出修改路径必须清晰。
- 检查结果要说明通过或失败。
- 需要人工复核的事项单独列出。

### 使用后人工检查点

- 查看实际 diff，确认只改了允许范围。
- 确认没有新增敏感信息或无关文件。
- 检查轻量测试或文档检查是否真的执行。
- 对 Agent 未确认的高风险操作保持拒绝。

## 通用人工复核清单

每次使用上述模板后，建议额外做一次通用复核：

- 是否包含真实密钥、密码、cookie、token、生产连接串或用户隐私数据。
- 是否有 AI 编造的业务规则、接口字段、错误码或指标。
- 是否有高风险命令，例如删除、覆盖、强推、直接改生产数据。
- 是否明确了验收标准和验证方式。
- 是否保留了足够上下文，方便其他同学复盘。
