# 常见开发中如何使用 AI / LLM / Agent

这份指南面向刚开始把 AI 用到开发流程的小白。目标不是让 AI 替你负责，而是让它成为“会提问、会解释、会辅助验证的开发搭档”。

使用原则：

- 先给背景，再给任务，不要只丢一句“帮我改”。
- 让 AI 输出计划、风险和验证方式，再决定是否执行。
- 不粘贴真实 API Key、token、cookie、生产连接串、用户隐私数据。
- 对代码改动、数据库变更、权限变更、线上操作始终人工复核。
- AI 的回答是候选方案，不是事实来源；关键结论要用代码、测试、日志或文档验证。

## 场景 1：需求澄清

### 适合用 AI 做什么

- 把一句模糊需求拆成背景、目标、用户、输入、输出、边界和验收标准。
- 找出缺失问题，例如权限、异常、性能、兼容性、数据来源。
- 生成需求澄清清单，方便和产品、业务或同事确认。

### 不适合做什么

- 不要让 AI 直接替业务拍板。
- 不要把敏感用户原始数据贴给 AI。
- 不要把 AI 猜测的需求当作已确认事实。

### 推荐 Prompt 模板

```text
你是资深需求分析师。请帮我澄清下面这个开发需求。

背景：<项目背景>
原始需求：<需求原文>
当前系统：<已有能力或限制>

请输出：
1. 你理解的目标
2. 需要确认的问题
3. 输入、输出、边界条件
4. 权限、安全、异常场景
5. 可验收标准
6. 最小可实现版本

不要编造业务事实；不确定的地方请标记为“待确认”。
```

### 检查清单

- [ ] 是否区分了“已知事实”和“待确认问题”？
- [ ] 是否有清晰验收标准？
- [ ] 是否覆盖失败场景和权限边界？
- [ ] 是否能拆出最小版本？

### 常见错误

- 只让 AI “整理需求”，但没有给背景。
- 没有让 AI 明确待确认问题。
- 把 AI 补出来的业务规则直接当真。

## 场景 2：技术方案设计

### 适合用 AI 做什么

- 对比多种实现方案。
- 列出模块边界、数据流、错误处理和测试策略。
- 识别安全、性能、兼容性风险。

### 不适合做什么

- 不要让 AI 在不了解现有架构时直接选型。
- 不要跳过团队规范、成本、运维约束。

### 推荐 Prompt 模板

```text
你是资深后端/全栈工程师。请基于下面信息设计技术方案。

需求：<需求>
现有系统：<模块、接口、数据库、限制>
约束：<时间、性能、安全、兼容性>

请输出：
1. 推荐方案
2. 备选方案和取舍
3. 模块拆分
4. 数据流和接口草案
5. 错误处理
6. 测试计划
7. 风险和需要人工确认的问题
```

### 检查清单

- [ ] 是否说明为什么推荐该方案？
- [ ] 是否有不采用其他方案的理由？
- [ ] 是否包含测试和回滚策略？
- [ ] 是否符合项目现有技术栈？

### 常见错误

- 让 AI 设计过度复杂架构。
- 没有告诉 AI 当前项目已有约定。
- 没有让 AI 列出风险。

## 场景 3：读陌生代码

### 适合用 AI 做什么

- 总结文件职责、核心函数、调用链。
- 找入口、配置、数据模型和测试位置。
- 把复杂代码解释成学习笔记。

### 不适合做什么

- 不要只贴一小段代码就要求解释整个系统。
- 不要把 AI 对调用链的猜测当作事实；要用搜索和测试验证。

### 推荐 Prompt 模板

```text
你是代码讲解老师。请帮我理解这段代码。

我的背景：<例如 Python 初学者 / 会后端但不熟前端>
目标：<我想理解它如何处理登录 / 如何调用工具>
代码：
```<语言>
<代码片段>
```

请输出：
1. 这段代码解决什么问题
2. 关键函数/类的职责
3. 数据从哪里来、到哪里去
4. 可能依赖的其他文件
5. 我下一步应该看哪些位置
6. 用初学者能懂的话解释一次
```

### 检查清单

- [ ] 是否知道入口函数在哪里？
- [ ] 是否理解输入和输出？
- [ ] 是否知道相关测试在哪里？
- [ ] 是否能自己复述一遍？

### 常见错误

- 只问“这段代码什么意思”，没有说明学习目标。
- 不验证 AI 说的文件路径或函数名。
- 没有结合实际运行结果。

## 场景 4：定位 Bug

### 适合用 AI 做什么

- 根据错误日志推测可能原因。
- 给出最小复现步骤。
- 建议排查顺序。
- 帮你设计回归测试。

### 不适合做什么

- 不要让 AI 直接大范围重构。
- 不要省略完整错误信息、触发条件和最近变更。

### 推荐 Prompt 模板

```text
你是调试助手。请帮我分析下面的 bug。

现象：<用户看到什么>
期望：<应该发生什么>
触发步骤：<如何复现>
错误日志：<脱敏后的日志>
最近变更：<相关提交或改动>
相关代码：<代码片段>

请输出：
1. 最可能的 3 个原因，按概率排序
2. 每个原因如何验证
3. 最小修复建议
4. 应补充的测试用例
5. 哪些信息不足，需要我继续提供
```

### 检查清单

- [ ] 是否能稳定复现？
- [ ] 是否定位到具体边界条件？
- [ ] 是否先写或补了测试？
- [ ] 修复是否足够小？

### 常见错误

- 只贴报错最后一行。
- 没有提供输入数据和环境。
- 让 AI 顺手改太多无关代码。

## 场景 5：写测试

### 适合用 AI 做什么

- 从需求生成测试用例清单。
- 补充正常、边界、异常、安全用例。
- 根据现有测试风格生成草稿。

### 不适合做什么

- 不要为了通过测试而修改测试期望。
- 不要只测 happy path。

### 推荐 Prompt 模板

```text
你是测试工程师。请帮我为下面功能设计测试。

功能说明：<功能>
输入输出：<输入输出>
边界条件：<已知边界>
现有测试风格：<贴一段已有测试>

请输出：
1. 测试用例表格：名称、输入、期望、覆盖点
2. 最重要的 3 个回归测试
3. 可能遗漏的异常场景
4. 如果要写成 unittest/pytest，给出代码草稿
```

### 检查清单

- [ ] 是否覆盖正常、边界、异常？
- [ ] 是否测试行为而不是实现细节？
- [ ] 是否能在本地稳定运行？
- [ ] 是否避免真实网络和真实密钥？

### 常见错误

- 测试只覆盖 AI 生成的样例输入。
- 断言过于宽松，失败也看不出来。
- 在测试里调用真实外部服务。

## 场景 6：重构

### 适合用 AI 做什么

- 识别重复逻辑和命名问题。
- 给出小步重构计划。
- 生成迁移前后的测试清单。

### 不适合做什么

- 不要让 AI 一次性大改全项目。
- 不要在没有测试保护时重构核心路径。

### 推荐 Prompt 模板

```text
你是重构教练。请帮我制定小步重构计划。

目标：<例如降低重复、拆分函数、改善命名>
代码范围：<文件/函数>
约束：<不能改变外部行为、不改接口等>
测试情况：<已有测试>

请输出：
1. 当前主要问题
2. 最小安全重构步骤
3. 每一步如何验证
4. 不建议现在做的改动
5. 重构后的风险点
```

### 检查清单

- [ ] 是否明确“不改变外部行为”？
- [ ] 是否每一步都能测试？
- [ ] 是否避免顺手加功能？
- [ ] 是否保留必要兼容？

### 常见错误

- 借重构之名改需求。
- 变量命名换了一遍但行为没验证。
- 删除看似无用但实际被动态调用的代码。

## 场景 7：Code Review

### 适合用 AI 做什么

- 从 diff 中找潜在 bug、安全风险、遗漏测试。
- 检查是否符合需求和项目风格。
- 帮你写 review checklist。

### 不适合做什么

- 不要让 AI 只做“夸奖式 review”。
- 不要把 AI 的 review 当作最终审批。

### 推荐 Prompt 模板

```text
你是严格但建设性的代码审查者。请审查下面 diff。

需求背景：<为什么改>
重点关注：正确性、安全、边界、测试、可维护性
Diff：
```diff
<diff>
```

请输出：
1. 必须修复的问题
2. 建议优化的问题
3. 测试缺口
4. 安全或权限风险
5. 如果没有问题，请说明你检查了哪些方面
```

### 检查清单

- [ ] 是否检查边界条件？
- [ ] 是否检查安全风险？
- [ ] 是否检查测试是否足够？
- [ ] 是否避免主观风格争论？

### 常见错误

- 只让 AI 看代码，不给需求背景。
- 没有区分 must fix 和 nice to have。
- 不复核 AI 指出的“问题”是否真实存在。

## 场景 8：写文档和注释

### 适合用 AI 做什么

- 把实现整理成 README、接口说明、使用示例。
- 把复杂流程写成步骤和 FAQ。
- 帮你生成变更说明。

### 不适合做什么

- 不要让 AI 编造不存在的参数、命令或功能。
- 不要写解释“代码做了什么”的冗余注释。

### 推荐 Prompt 模板

```text
你是技术文档作者。请基于下面事实写文档。

目标读者：<初学者/开发者/运维>
真实功能：<列出功能事实>
运行方式：<命令>
限制：<已知限制>

请输出：
1. 简短介绍
2. 快速开始
3. 参数说明
4. 常见问题
5. 安全注意事项

只能基于我提供的事实写，不要编造功能。
```

### 检查清单

- [ ] 文档中的命令真实可运行？
- [ ] 链接是否存在？
- [ ] 是否说明限制？
- [ ] 是否没有暴露敏感信息？

### 常见错误

- AI 编造配置项。
- 文档和代码不同步。
- 示例里出现真实密钥或生产地址。

## 场景 9：Git、Commit、PR

### 适合用 AI 做什么

- 总结 diff，生成 commit message 草稿。
- 写 PR 描述、测试计划、风险说明。
- 帮你检查是否混入无关文件。

### 不适合做什么

- 不要让 AI 自动 push、force push、改历史，除非你明确确认。
- 不要提交 `.env`、密钥、临时文件。

### 推荐 Prompt 模板

```text
请根据下面 git diff 帮我写提交说明和 PR 描述。

变更目标：<为什么改>
Diff 摘要：<diff 或文件列表>
验证结果：<测试命令和结果>

请输出：
1. 一句话 commit message
2. PR Summary bullet list
3. Test plan
4. 风险和回滚方式
5. 是否有不应提交的文件迹象
```

### 检查清单

- [ ] `git status` 是否只有预期文件？
- [ ] 是否运行测试？
- [ ] PR 描述是否说明 why？
- [ ] 是否没有真实密钥和大文件？

### 常见错误

- commit message 只写“update”。
- 把格式化、重构、功能混在一个巨大 PR。
- 没有写测试计划。

## 场景 10：CI 失败分析

### 适合用 AI 做什么

- 解释 CI 日志。
- 判断失败是测试、依赖、环境还是格式问题。
- 给出本地复现命令。

### 不适合做什么

- 不要直接跳过 CI 或禁用测试。
- 不要用 `--no-verify` 逃避问题。

### 推荐 Prompt 模板

```text
你是 CI 排障助手。请分析下面 CI 失败。

失败步骤：<step name>
日志：<脱敏日志>
最近改动：<文件或 diff 摘要>
本地环境：<Python/Node/OS 等>

请输出：
1. 失败类型
2. 最可能根因
3. 本地复现命令
4. 修复步骤
5. 如何确认修复成功
```

### 检查清单

- [ ] 是否能本地复现？
- [ ] 是否修复根因而不是绕过？
- [ ] 是否更新必要测试或文档？
- [ ] 是否确认 CI 相关配置没有误改？

### 常见错误

- 只看最后一行错误。
- 误以为 CI 环境和本机完全一样。
- 为了通过 CI 删除测试。

## 场景 11：RAG 知识库开发

### 适合用 AI 做什么

- 设计文档切分策略。
- 生成检索评测问题。
- 分析回答是否有引用、是否幻觉。

### 不适合做什么

- 不要把未脱敏的内部文档发给外部模型。
- 不要只看回答流畅度，不看引用来源。

### 推荐 Prompt 模板

```text
你是 RAG 系统设计助手。请帮我设计知识库问答流程。

文档类型：<Markdown/PDF/FAQ>
用户问题类型：<常见问题>
约束：<隐私、权限、延迟>

请输出：
1. chunk 策略
2. metadata 字段
3. 检索和重排流程
4. 回答引用格式
5. 评测问题集设计
6. 失败时如何拒答
```

### 检查清单

- [ ] 回答是否有来源引用？
- [ ] 检索结果是否真的包含答案？
- [ ] 是否能处理找不到答案？
- [ ] 是否有权限和脱敏策略？

### 常见错误

- chunk 太大或太小。
- 引用看起来存在但无法定位原文。
- 模型在检索不到时仍然编答案。

## 场景 12：MCP 工具接入

### 适合用 AI 做什么

- 设计 MCP tool/resource/prompt。
- 检查 JSON-RPC 请求和响应形状。
- 生成工具 schema 和错误码建议。

### 不适合做什么

- 不要把高风险操作直接暴露成无确认工具。
- 不要让工具返回未脱敏敏感数据。

### 推荐 Prompt 模板

```text
你是 MCP Server 设计助手。请帮我把下面能力设计成 MCP 工具。

能力描述：<要暴露的能力>
调用者：<Claude Desktop/Claude Code/内部 Agent>
风险：<读/写/删除/外部请求>

请输出：
1. tool 名称和 description
2. input schema
3. 成功响应结构
4. 错误响应结构
5. 权限确认规则
6. 调试用 JSON-RPC 示例
```

### 检查清单

- [ ] tool description 是否清晰？
- [ ] schema 是否稳定？
- [ ] 高风险动作是否需要确认？
- [ ] 日志是否脱敏？

### 常见错误

- 工具名太泛，例如 `run`、`do_task`。
- schema 不约束输入。
- 错误响应不结构化，Agent 难以恢复。

## 场景 13：Claude Code / Skill 日常用法

### 适合用 AI 做什么

- 让 Claude Code 读项目、制定计划、改代码、跑测试。
- 把重复流程沉淀成 Skill。
- 用 hooks 做快速、低风险检查。
- 用 subagents 并行做搜索、审查、文档补充。

### 不适合做什么

- 不要让 Agent 直接执行高风险命令。
- 不要在没有计划时让它大范围改代码。
- 不要把生产凭证放进 prompt 或文件。

### 推荐 Prompt 模板

```text
请先只读分析，不要改文件。

目标：<我要完成什么>
范围：<相关目录/文件>
约束：<不能改什么、必须保留什么>
验证方式：<测试命令或验收方式>

请输出：
1. 你会先看哪些文件
2. 可能的实现方案
3. 风险点
4. 最小修改计划
5. 需要我确认的问题
```

### 检查清单

- [ ] 是否先探索再实现？
- [ ] 是否明确修改范围？
- [ ] 是否运行测试？
- [ ] 是否总结风险和后续工作？

### 常见错误

- 一开始就要求“直接改”。
- 没有说明测试命令。
- 没有区分当前任务和顺手优化。

## 场景 14：什么时候不要用 AI

下面情况不建议直接使用 AI，或必须先脱敏、隔离、确认：

- 涉及真实密钥、生产数据库、用户隐私、合同、财务、医疗等敏感信息。
- 需要精确法律、合规、安全结论。
- 要执行删除、覆盖、force push、生产发布、数据库迁移等高风险操作。
- 你无法验证 AI 输出是否正确。
- 团队明确要求某类代码或数据不能进入外部工具。

如果必须使用，先做：

1. 脱敏。
2. 缩小范围。
3. 要求 AI 标记不确定点。
4. 人工复核。
5. 用测试、日志、文档验证。

## 日常开发使用 AI 的推荐流程

```text
1. 说明目标和背景
2. 要求 AI 先提问题或制定计划
3. 人工确认范围
4. 小步执行
5. 运行测试或检查
6. 让 AI 总结改动、风险、下一步
7. 人工 review 后再提交
```

## 最小安全检查清单

每次使用 AI 辅助开发前后，至少检查：

- [ ] 没有粘贴真实 API Key、token、cookie、生产连接串。
- [ ] 没有粘贴未脱敏用户隐私或生产数据。
- [ ] AI 输出中的文件名、函数名、命令都经过验证。
- [ ] 高风险操作需要人工确认。
- [ ] 代码改动有测试或手动验证。
- [ ] 文档和 PR 说明没有编造功能。
- [ ] 不确定结论已标记为待确认。
