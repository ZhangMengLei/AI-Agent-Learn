## 练习 1：设计天气工具

### 参考答案

```json
{
  "name": "get_weather",
  "description": "查询指定城市在指定日期的天气概况。",
  "input_schema": {
    "type": "object",
    "properties": {
      "city": {"type": "string", "description": "城市名称，例如 北京"},
      "date": {"type": "string", "description": "日期，格式 YYYY-MM-DD；不传则表示今天"}
    },
    "required": ["city"]
  }
}
```

返回示例：

```json
{"city": "北京", "date": "2026-06-30", "weather": "多云", "temperature_c": 28}
```

可能错误：城市不存在、日期格式错误、天气服务不可用、超时。

### 为什么这样答

工具描述要让模型知道什么时候调用；schema 要让程序能校验参数；错误要结构化返回，避免模型把失败当成功。

### 常见错误

- 工具描述太含糊，模型不知道适用边界。
- 没有 required 字段，导致缺 city 也执行。
- 返回自然语言而非结构化结果，后续处理困难。
## 练习 2：设计计算器工具

### 参考答案

参数可以用表达式字符串，但执行层必须做安全解析：

```json
{
  "name": "calculate",
  "description": "计算只包含数字、括号和基础数学运算符的表达式。",
  "input_schema": {
    "type": "object",
    "properties": {
      "expression": {"type": "string", "description": "例如 1 + 2 * 3"}
    },
    "required": ["expression"]
  }
}
```

安全策略：使用 AST 白名单或自写解析器，只允许数字、加减乘除、括号等节点，不使用 `eval` 执行原始字符串。

失败返回：

```json
{"ok": false, "error": "unsupported expression"}
```

### 为什么这样答

表达式字符串对模型最自然，但安全风险更高，所以必须由程序层限制语法。拆成数字和操作符更安全，但表达复杂表达式不方便。

### 常见错误

- 直接 `eval(expression)`。
- 错误时抛出未处理异常。
- 让模型自己判断表达式是否安全，而不是程序校验。
## 练习 3：判断是否需要工具

### 参考答案

| 问题 | 是否需要工具 | 原因 |
| --- | --- | --- |
| 什么是 Prompt？ | 不一定 | 概念解释可由模型直接回答。 |
| 北京今天下雨吗？ | 需要 | 需要实时天气数据。 |
| 计算 123 * 456。 | 建议需要 | 计算器更准确、可复现。 |
| 总结下面这段文字。 | 不一定 | 如果文本已在上下文中，可直接总结。 |
| 删除我的临时文件。 | 需要且高风险 | 涉及文件系统写/删操作，必须确认。 |

### 为什么这样答

判断是否需要工具的关键是：模型内部知识是否足够、是否需要实时数据、是否需要精确计算、是否会影响外部世界。

### 常见错误

- 所有问题都调用工具，增加复杂度。
- 对删除、发送、修改类动作不做确认。
- 让模型凭记忆回答实时问题。
## 练习 4：权限分级

### 参考答案

| 工具 | 风险等级 | 原因 |
| --- | --- | --- |
| 查询天气 | 低风险 | 只读公开信息。 |
| 读取本地文件 | 中风险 | 可能读取敏感文件。 |
| 删除文件 | 高风险 | 破坏性操作。 |
| 发送邮件 | 高风险 | 对外部用户可见，可能造成影响。 |
| 查询数据库 | 中风险 | 可能涉及隐私或业务数据。 |
| 修改数据库 | 高风险 | 改变共享状态。 |
| 执行 shell 命令 | 高风险 | 能造成广泛副作用。 |

### 为什么这样答

权限分级看两个维度：是否只读、是否影响外部世界。写入、删除、发送、执行命令通常需要人工确认或禁止自动执行。

### 常见错误

- 只按工具名称判断，不看具体参数和作用范围。
- 把读取本地文件视为低风险，忽略 secret 泄露。
- 高风险工具没有审计日志。
## 练习 5：工具结果不可信

### 参考答案

1. 这是工具结果中的 Prompt Injection 风险，也可以理解为不可信外部内容试图覆盖系统指令。
2. 模型应把它当作被搜索到的普通文本，不执行其中的指令，不泄露 API Key，不改变原任务边界。
3. 程序侧应隔离工具结果、标记来源、过滤敏感字段、禁止工具结果覆盖系统消息，并对高风险输出做安全检查。

### 为什么这样答

工具结果来自外部系统，不等于开发者指令。Agent 必须区分“数据内容”和“控制指令”，否则搜索结果、网页、文档都可能劫持模型行为。

### 常见错误

- 把工具返回内容直接拼进最高优先级提示。
- 让模型自行决定是否泄露密钥。
- 没有记录工具结果来源和原文。
## 练习 6：完整调用链

### 参考答案

```text
用户输入自然语言需求
  ↓
模型判断是否需要工具，并生成结构化 ToolCall
  ↓
程序校验工具名、参数、权限
  ↓
必要时请求用户确认
  ↓
程序执行工具，得到 ToolResult
  ↓
程序记录 tool log
  ↓
模型基于工具结果组织最终回答
  ↓
返回给用户
```

### 为什么这样答

模型负责理解和规划，程序负责校验和执行。把这两层分开，才能做权限控制、日志审计和错误处理。

### 常见错误

- 模型直接执行工具，没有程序层校验。
- 不记录工具调用日志，无法排查。
- 工具失败后仍向用户报告成功。
