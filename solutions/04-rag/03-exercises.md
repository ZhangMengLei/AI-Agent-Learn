# RAG 知识库问答 练习参考答案

对应题目：[lessons/04-rag/03-exercises.md](../../lessons/04-rag/03-exercises.md)

这些答案用于做完练习后的复盘。请重点关注“为什么这样答”和“常见错误”。

## 练习 1：文档切分

### 参考答案

1. Markdown 文档优先按标题切，再按段落或长度补切。标题能保留语义结构，长度限制能避免 chunk 过大。
2. 入门项目可从 300-800 中文字或约 500 token 左右开始，根据召回效果调整。
3. 需要少量 overlap，尤其是跨段落定义、步骤说明和上下文依赖强的文档；可从 50-100 token 开始。
4. metadata 至少包含 `document_id`、`source`、`title`、`section`、`chunk_id`、`position`、`updated_at`。

### 为什么这样答

RAG 的 chunk 既要足够小，方便精确召回；又要足够完整，避免回答缺上下文。metadata 决定了后续引用、过滤和排查能力。

### 常见错误

- 只按固定长度切，破坏标题和段落语义。
- chunk 太大导致召回不精确。
- 不保存来源信息，答案无法引用。
## 练习 2：设计 chunk 结构

### 参考答案

```json
{
  "chunk_id": "prompt-basic-003",
  "document_id": "prompt-basic",
  "title": "Prompt 基础",
  "content": "Prompt 是用户给模型的任务说明...",
  "source": "data/docs/prompt-basic.md",
  "section": "什么是 Prompt",
  "metadata": {
    "paragraph_index": 3,
    "start_char": 120,
    "end_char": 260
  }
}
```

### 为什么这样答

题目要求的字段用于回答生成和引用；额外 metadata 用于调试、排序和定位原文。`chunk_id` 应稳定，便于日志和评测复现。

### 常见错误

- `content` 中混入过多无关段落。
- `source` 只写文件名，不够定位。
- chunk_id 每次随机生成，导致评测结果不可复现。
## 练习 3：检索问题改写

### 参考答案

原问题：`这个东西怎么用？`

结合上下文“用户正在学习 MCP”，可改写为：

```text
MCP 的基本使用方式是什么？MCP Client 如何连接 MCP Server，并调用 Server 暴露的 Tool、Resource 或 Prompt？
```

也可以拆成检索关键词：

```text
MCP 使用方式、MCP Client、MCP Server、Tool、Resource、Prompt、配置
```

### 为什么这样答

原问题缺少指代对象，“这个东西”无法检索。改写时要补足主题、核心实体和可能出现于文档中的关键词。

### 常见错误

- 只把问题改成“怎么用 MCP”，关键词仍然不足。
- 改写时加入用户没问的目标。
- 用过度口语化表达，和文档术语不匹配。
## 练习 4：判断检索质量

### 参考答案

检索结果 A 更相关。用户问的是 Agent 和 Chatbot 的区别，A 提到 Agent 的目标、工具调用和自主执行，直接覆盖区别的核心维度。B 介绍 Transformer 注意力机制，属于模型底层原理，不回答该问题。

### 为什么这样答

检索质量要看结果是否能支持最终回答，而不是看内容是否“也和 AI 有关”。相关片段应覆盖问题中的核心实体和比较维度。

### 常见错误

- 只要包含 AI 关键词就认为相关。
- 忽略用户问题是“比较题”。
- 不区分背景知识和直接证据。
## 练习 5：设计回答约束

### 参考答案

```text
你是一个基于检索资料回答问题的助手。请严格遵守：

1. 只使用【检索资料】中的信息回答，不使用资料外知识。
2. 如果资料不足以回答，请明确说“根据当前资料无法确定”。
3. 每个关键结论后标注来源 ID，例如 [C1]。
4. 回答末尾列出“引用来源”。
5. 不要编造来源，不要引用没有支持该结论的片段。

【用户问题】
{{question}}

【检索资料】
{{chunks}}
```

### 为什么这样答

RAG 的目标不是让模型自由发挥，而是让模型基于证据回答。来源 ID 和拒答规则能减少幻觉，也方便人工检查。

### 常见错误

- 只要求“参考资料”，没有禁止资料外知识。
- 要求列来源但不要求结论和来源对应。
- 检索不足时仍强行回答。
## 练习 6：RAG 失败分析

### 参考答案

| 现象 | 可能原因 | 排查方向 |
| --- | --- | --- |
| 检索不到相关内容 | 文档没入库、切分差、关键词不匹配、embedding 质量差 | 检查索引、query rewrite、召回日志 |
| 检索到了但模型回答错误 | prompt 约束弱、上下文太长、证据冲突 | 检查回答 prompt、top-k、引用一致性 |
| 回答没有引用来源 | prompt 未强制引用、chunk 没有 ID | 增加来源格式和输出校验 |
| 引用了错误来源 | chunk 排序错误、模型随意引用 | 校验引用 ID 是否真的支持结论 |
| 回答太长或太短 | 输出格式和长度未约束 | 设置回答结构、字数范围、重点数量 |

### 为什么这样答

RAG 问题可能发生在入库、检索、重排、生成、引用校验多个环节。排查时要看中间结果，而不是只看最终答案。

### 常见错误

- 只调 Prompt，不看召回结果。
- 只调 chunk size，不看问题改写。
- 没有黄金集，无法判断调整是否变好。
