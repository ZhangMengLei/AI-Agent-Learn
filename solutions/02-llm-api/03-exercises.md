# LLM API 与 SDK 练习参考答案

对应题目：[lessons/02-llm-api/03-exercises.md](../../lessons/02-llm-api/03-exercises.md)

这些答案用于做完练习后的复盘。请重点关注“为什么这样答”和“常见错误”。

## 练习 1：理解请求结构

### 参考答案

一次基础 LLM API 请求通常包含：

```json
{
  "model": "claude-opus-4-7",
  "messages": [
    {"role": "user", "content": "请解释什么是 Agent"}
  ],
  "temperature": 0.3,
  "max_tokens": 800,
  "stream": false
}
```

### 为什么这样答

`model` 决定使用哪个模型；`messages` 承载对话内容；`temperature` 控制随机性；`max_tokens` 控制最大输出长度；`stream` 决定是否边生成边返回。

### 常见错误

- 把 API Key 写进请求体或代码文件。
- 不限制 `max_tokens`，导致成本和输出不可控。
- 混淆 system/developer/user 等不同层级消息的职责。
## 练习 2：设计 messages

### 参考答案

```json
[
  {
    "role": "system",
    "content": "你是一名耐心的 AI 学习助手。请用中文、用初学者能理解的方式解释概念，先给定义，再给例子，避免夸大能力。"
  },
  {
    "role": "user",
    "content": "请解释什么是 Agent。"
  }
]
```

### 为什么这样答

system message 设定长期行为和回答风格，user message 表达当前具体任务。把角色、语言、受众和边界放在 system 中，比只在 user 中临时说明更稳定。

### 常见错误

- system message 写得太空泛，例如“你很厉害”。
- user message 同时塞入大量长期规则，导致多轮复用困难。
- 没有说明受众，回答可能过深或过浅。
## 练习 3：多轮对话

### 参考答案

```json
[
  {"role": "user", "content": "什么是 Prompt？"},
  {"role": "assistant", "content": "Prompt 是你给 AI 的任务说明，它告诉模型要做什么、按什么要求输出。"},
  {"role": "user", "content": "那 System Prompt 是什么？"}
]
```

第三次请求需要带上前面的历史，因为“那”指向上一轮的 Prompt 话题。如果不带历史，模型只能猜测上下文，可能无法理解追问对象。

### 为什么这样答

LLM API 通常是无状态的。服务端不会自动记住你之前说了什么，应用需要把必要历史重新发送给模型。

### 常见错误

- 以为模型天然记得所有历史。
- 把全部历史无限追加，导致成本变高、上下文超限。
- 不做摘要或裁剪，长期对话质量会下降。
## 练习 4：流式输出

### 参考答案

```text
初始化完整回答为空字符串
发起 stream=true 的请求
for 每个返回片段 chunk:
    提取 chunk 中的新文本 delta
    立即打印 delta
    追加 delta 到完整回答
请求结束后:
    把完整回答保存到对话历史
    记录 token、耗时和状态
```

### 为什么这样答

流式输出改善的是首字响应和交互体验，不改变模型实际需要生成的内容。最终仍要拼接完整结果，才能保存历史或做后续处理。

### 常见错误

- 只打印片段，不保存完整回答。
- 把每个 chunk 都当成完整 JSON 回答。
- 认为流式一定省钱；它通常只是更快显示。
## 练习 5：调用日志

### 参考答案

```json
{
  "request_id": "req_20260630_001",
  "created_at": "2026-06-30T10:00:00Z",
  "model": "claude-opus-4-7",
  "input_tokens": 1200,
  "output_tokens": 300,
  "latency_ms": 1850,
  "success": true,
  "error": null
}
```

### 为什么这样答

这组字段能支持排障、成本统计、延迟分析和失败率统计。日志应记录指标和状态，不应默认记录完整密钥、隐私数据或敏感原文。

### 常见错误

- 记录完整 API Key、cookie 或私密用户输入。
- 只有错误日志，没有成功请求日志，无法计算基线。
- 不记录 request_id，线上排查难以串联链路。
## 练习 6：成本意识

### 参考答案

1. 长上下文会增加成本，因为输入 token 越多，模型处理的内容越多，计费和延迟通常都会上升。
2. 流式输出不一定减少总成本，因为总生成 token 没有减少，只是更早把结果显示出来。
3. Prompt Caching 适合固定长前缀反复使用的场景，例如长系统提示、固定工具说明、长期不变的知识背景。
4. 多轮对话需要摘要或裁剪，因为上下文窗口有限，全部历史追加会增加成本并稀释重点。

### 为什么这样答

API 工程不只关注“能不能调用”，还要关注延迟、稳定性、成本和上下文质量。上下文管理是长期对话应用的核心能力。

### 常见错误

- 把所有历史无脑传入。
- 为短提示强行做缓存，收益不明显。
- 只限制输出长度，不控制输入长度。
