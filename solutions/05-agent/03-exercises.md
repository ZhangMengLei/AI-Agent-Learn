这些答案用于做完练习后的复盘。请重点关注“为什么这样答”和“常见错误”。

## 练习 1：区分 Chatbot 和 Agent

### 参考答案

| 场景 | 更适合 | 原因 |
| --- | --- | --- |
| 解释什么是 Prompt | Chatbot | 单轮知识解释即可。 |
| 自动查资料并写报告 | Agent | 需要搜索、读取、整理、生成多步行动。 |
| 翻译一句英文 | Chatbot | 明确输入输出，无需工具循环。 |
| 读取项目、修改代码、运行测试 | Agent | 需要操作文件、执行命令、根据结果调整。 |
| 根据用户目标连续调用多个工具 | Agent | 核心就是计划和工具编排。 |

### 为什么这样答

Chatbot 偏回答，Agent 偏完成任务。判断标准是是否需要多步计划、工具调用、状态跟踪和结果反馈。

### 常见错误

- 把所有使用 LLM 的系统都叫 Agent。
- 忽略外部动作的权限和风险。
- 认为 Agent 一定比 Chatbot 更好。
## 练习 2：拆解目标

### 参考答案

目标：帮我学习 MCP，并整理一份适合初学者的笔记。

可拆为：

1. 明确学习范围：MCP 是什么、解决什么问题、核心组件有哪些。
2. 搜索或读取 MCP 入门资料，收集 Client、Server、Tool、Resource、Prompt 等概念。
3. 将资料整理成初学者能懂的结构：概念、类比、示例、常见误区。
4. 生成一份 Markdown 笔记，包含目录、重点和实践建议。
5. 检查笔记是否准确、是否有术语未解释、是否适合初学者。

### 为什么这样答

好的目标拆解要从“搜集资料”到“整理结构”再到“质量检查”。只写“查资料、写笔记”过粗，Agent 不容易执行和判断完成。

### 常见错误

- 步骤不可执行，例如“深入理解 MCP”。
- 没有验收标准。
- 没有把初学者视角放进任务。
## 练习 3：设计 Agent 循环

### 参考答案

```text
state = {goal, notes, tool_log, iteration: 0}
plan = make_plan(goal)

while iteration < max_iterations:
    act = choose_next_action(plan, state)
    result = execute_tool(act)
    observe = summarize_result(result)
    state.notes.append(observe)
    reflect = decide_continue_or_stop(goal, state, observe)
    if reflect.finished:
        break
    iteration += 1

return finish(state.notes, tool_log, stop_reason)
```

### 为什么这样答

循环中必须有目标、计划、行动、观察、反思和终止。`max_iterations` 是硬边界，防止 Agent 因错误计划或重复搜索无限运行。

### 常见错误

- 只有 plan 和 act，没有 observe/reflect。
- 没有状态，下一步不知道前面做过什么。
- 没有最大迭代次数。
## 练习 4：设计工具集

### 参考答案

| 工具 | 描述 | 输入参数 | 风险等级 |
| --- | --- | --- | --- |
| `search_materials` | 按关键词搜索资料列表 | `query`, `top_k` | 低到中，取决于数据源 |
| `read_document` | 读取指定文档内容 | `document_id` 或 `path` | 中，可能读取敏感内容 |
| `save_note` | 保存结构化学习笔记 | `title`, `content`, `tags` | 中，写入本地文件或数据库 |

### 为什么这样答

研究助手需要“找资料、读资料、沉淀结果”三类能力。工具数量不必多，但职责要清晰，读写权限要分开。

### 常见错误

- 一个工具同时搜索、读取、写入，权限难控制。
- `read_document` 不限制路径范围。
- `save_note` 自动覆盖已有内容。
## 练习 5：终止条件

### 参考答案

1. 连续 N 次搜索没有新增有效资料时停止，例如连续 2 次新增信息为空。
2. 已覆盖计划中的全部关键问题时停止，例如定义、组件、使用流程、风险都已有答案。
3. 达到 `max_iterations` 或时间/成本预算时停止，并报告未完成部分。

### 为什么这样答

Agent 需要质量终止条件和资源终止条件。只靠模型判断“差不多了”不稳定，必须有可计算边界。

### 常见错误

- 没有记录已见资料，导致重复搜索。
- 到达上限时不说明原因。
- 停止条件只看轮数，不看目标覆盖度。
## 练习 6：权限判断

### 参考答案

| 动作 | 是否需要确认 | 原因 |
| --- | --- | --- |
| 搜索公开资料 | 通常不需要 | 只读公开信息。 |
| 读取当前项目 README | 通常不需要 | 低风险项目文件。 |
| 修改代码文件 | 需要 | 会改变工作区。 |
| 删除文件 | 必须确认 | 破坏性操作。 |
| 提交 git commit | 需要 | 会改变版本历史。 |
| push 到远程仓库 | 必须确认 | 影响共享远程状态。 |

### 为什么这样答

权限判断要看可逆性和影响范围。本地只读通常风险低；写文件、删除、提交、推送都可能影响用户工作或团队协作。

### 常见错误

- 认为 commit/push 都只是本地操作。
- 删除文件前不检查是否是用户未提交工作。
- 把所有读文件都视为安全，忽略敏感路径。
## 练习 7：反思机制

### 参考答案

```text
你是 Agent 的反思模块。请基于目标、上一步动作和工具结果做判断。

目标：{{goal}}
上一步动作：{{action}}
工具结果：{{observation}}
当前已知信息：{{notes}}

请输出：
1. 当前结果是否有用：有用 / 部分有用 / 无用。
2. 对目标有什么新增帮助。
3. 是否需要继续：继续 / 停止。
4. 下一步动作建议。
5. 如果停止，说明停止原因。
```

### 为什么这样答

反思 Prompt 让 Agent 不只是机械执行计划，而是根据观察结果调整方向。输出字段固定后，程序可以读取继续/停止决策。

### 常见错误

- 反思只写“总结一下结果”，没有决策。
- 没有包含目标，导致判断偏离任务。
- 不要求停止原因，用户无法复盘。
