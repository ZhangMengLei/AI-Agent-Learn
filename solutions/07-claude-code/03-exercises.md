## 练习 1：识别适合 CLI Agent 的任务

### 参考答案

| 任务 | 是否适合 | 是否需要确认 |
| --- | --- | --- |
| 解释一个函数作用 | 适合 | 通常不需要，只读。 |
| 修改项目代码并运行测试 | 适合 | 修改前可按任务授权，关键改动需可审查。 |
| 删除远程数据库 | 不适合自动执行 | 高危破坏性操作，应禁止。 |
| 生成 PR 总结 | 适合 | 生成文本不需要；发布 PR 需要确认。 |
| 批量重构变量名 | 适合 | 修改多文件，需测试和 diff 审查。 |
| 给线上用户发通知 | 不应自动执行 | 对外可见，必须人工确认或走正式流程。 |

### 为什么这样答

CLI Agent 擅长读代码、改代码、运行本地验证和整理说明；涉及生产、删除、外发、共享状态的动作必须谨慎。

### 常见错误

- 把“能做”当成“应该自动做”。
- 修改代码后不运行测试。
- 对外发布内容不让用户确认。
## 练习 2：编写 CLAUDE.md

### 参考答案

```markdown
# Project Guide

## 项目简介
这是一个 AI Agent 学习工程，包含课程文档、实现 demo、测试和自查清单。

## 常用命令
- `make install`：安装依赖
- `make list`：查看课程和数据
- `make check`：运行编译和测试

## 测试方式
- Python 测试使用 `python -m unittest discover -s tests`
- 编译检查使用 `python -m compileall -q implementations tests`

## 代码规范
- 优先使用 Python 标准库
- 示例代码保持教学可读性
- 不引入无必要抽象

## 安全注意事项
- 不提交 `.env` 或真实 API Key
- 写入、删除、push 等高风险动作需要确认
```

### 为什么这样答

CLAUDE.md 应给 Agent 足够的项目上下文和操作边界，让它知道怎么构建、测试、修改和避开风险。

### 常见错误

- 只写项目介绍，不写命令。
- 不说明安全边界。
- 写过期命令，导致 Agent 反复失败。
## 练习 3：设计 Slash Command

### 参考答案

命令：`/study-next`

输入：当前阶段、已完成内容、卡点，例如：

```text
/study-next stage=04-rag done=03-exercises stuck=chunking
```

执行步骤：

1. 读取当前阶段 README 和 checkpoint。
2. 判断下一步应该读理论、做练习、跑 demo 还是复盘。
3. 给出 3-5 个具体学习动作。
4. 标出需要产出的文件或笔记。

输出格式：

```text
当前阶段：
下一步任务：
建议时间：
完成标准：
如果卡住：
```

### 为什么这样答

好的 slash command 应把重复工作标准化，而不是变成另一个泛泛提问入口。输入和输出都固定，才容易长期使用。

### 常见错误

- 命令没有参数，无法结合当前进度。
- 输出没有完成标准。
- 让命令自动修改大量文件，不适合学习导航。
## 练习 4：设计 Hook

### 参考答案

触发时机：文件修改后，例如 Python 文件保存或 Edit/Write 后。

执行命令：

```bash
python -m compileall -q implementations tests
```

如果项目引入格式化工具，也可以运行对应 formatter，但应确保命令快速、可重复、不会修改无关文件。

失败处理：

1. 显示失败命令和错误摘要。
2. 阻止继续声称任务完成。
3. 让 Agent 修复语法或格式问题后重新验证。

### 为什么这样答

Hook 适合自动做低风险、快速、确定性的检查。格式化或编译失败时应反馈问题，而不是静默忽略。

### 常见错误

- Hook 执行慢命令，打断开发节奏。
- Hook 自动删除或覆盖文件。
- 失败后仍继续提交或报告成功。
## 练习 5：权限分类

### 参考答案

| 操作 | 分类 | 原因 |
| --- | --- | --- |
| 读取 README | 允许 | 只读低风险。 |
| 运行单元测试 | 允许 | 本地验证，通常可逆。 |
| 安装依赖 | 需要确认 | 会改变环境，可能耗时或引入风险。 |
| 删除文件 | 需要确认，重要文件应禁止自动 | 破坏性操作。 |
| git commit | 需要确认 | 改变版本历史。 |
| git push | 需要确认 | 影响远程共享状态。 |
| force push | 禁止自动执行 | 可能覆盖他人工作。 |
| 修改 CI 配置 | 需要确认 | 影响团队流水线。 |

### 为什么这样答

权限分类的核心是可逆性、影响范围和是否对外可见。越难回滚、影响越大，越需要人工控制。

### 常见错误

- 把所有 git 操作都当成本地安全操作。
- 允许 Agent 自动 force push。
- 修改 CI 后不解释影响。
## 练习 6：子 Agent 任务拆分

### 参考答案

可以按主题拆分：

| 子 Agent | 职责 |
| --- | --- |
| Prompt Agent | 生成 Prompt 基础、模板、练习和答案。 |
| API Agent | 生成 LLM API、流式、日志、成本相关材料。 |
| Tool/RAG Agent | 生成工具调用和 RAG 工程材料。 |
| Agent/MCP Agent | 生成 Agent 架构、MCP Server/Client 材料。 |
| Eval/Security Agent | 生成评测、安全、权限、上线检查材料。 |
| Reviewer Agent | 统一检查目录、术语、链接和重复内容。 |

### 为什么这样答

并行生成适合按主题拆分，但必须有统一结构和最终审校。否则多个 Agent 容易写出风格不一致、路径不一致的内容。

### 常见错误

- 每个 Agent 自定义目录结构。
- 没有 Reviewer 统一术语和链接。
- 让一个 Agent 同时负责太多不相关主题。
