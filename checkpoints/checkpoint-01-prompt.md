# Checkpoint 01：Prompt Engineering

## 概念自查

1. Prompt 为什么不应该只写成一句简单问题？
2. 角色、任务、上下文、约束、输出格式、验收标准分别解决什么问题？
3. 什么情况下应该要求模型先提问，而不是直接回答？
4. Few-shot 示例适合解决什么类型的问题？
5. 系统提示、用户提示和示例输入之间应该如何分工？

## 实践自查

1. 能否把一个模糊需求改写成结构化 Prompt？
2. 能否为同一个任务设计“简洁版”和“严格版”两个 Prompt？
3. 能否让模型输出稳定的 JSON、Markdown 表格或分步骤计划？
4. 能否识别 Prompt 中可能导致幻觉的表述？
5. 能否为 Prompt 增加“不知道就说明不确定”的约束？

## 工程自查

1. Prompt 是否可以被版本管理和复用？
2. Prompt 中是否混入了真实密钥、隐私数据或不可公开业务信息？
3. 是否为关键 Prompt 准备了测试用例？
4. 是否记录了模型、温度、上下文长度等影响结果的配置？
5. 是否有办法比较 Prompt 修改前后的效果？
